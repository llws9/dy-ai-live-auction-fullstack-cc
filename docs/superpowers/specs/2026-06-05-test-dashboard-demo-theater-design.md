# 独立测试平台演示剧场设计

## 1. 背景

当前 `frontend/test-dashboard` 已具备 `E2E`、`UserJourney`、`Screen`、`History`、`Report` 等页面，但面向评委演示时存在一个根本问题：页面能证明“测试跑过”，却不能让非技术观众直观看到“直播竞拍业务正在发生、系统如何保证结果可信”。

现状主要问题：

- `E2E` 页面偏技术控制台，展示参数、进度、步骤时间轴和最终 ID，观众难以建立业务因果。
- `UserJourney` 已覆盖更完整的买家链路，但表现形式仍偏验收表单和证据报告。
- `/test/screen` 当前是历史任务统计大屏，适合展示测试数量和通过率，不适合讲清一次实时业务闭环。

因此，本设计不新增一个相似测试页，而是复用并升级 `/test/screen`：把它从“历史统计大屏”改造成面向评委的“一键 UserJourney 演示剧场”。

## 2. 目标

面向评委/观众，在 30 秒内讲清三件事：

1. 系统自动创建并运行了一场真实业务语义的直播竞拍。
2. 直播间状态、出价、点天灯、订单、库存等业务事实在实时变化。
3. 最终结果证明业务闭环成立，并发结果唯一，资产状态一致。

成功标准：

- 演示人员只需要打开 `/test/screen` 并点击“开始演示”。
- 主屏不暴露 `seller_id`、`bidder_ids`、`duration` 等测试参数。
- 主屏优先展示直播间主视觉、当前价、领先者、出价事件、点天灯、订单生成、库存变化。
- 技术证据不丢失，但放到二级信息：报告链接、失败展开、步骤断言下钻。
- 原 `E2E` 与 `UserJourney` 页面继续作为技术控制台，不承担评委主演示。

## 3. 非目标

- 不重写测试服务架构。
- 不把所有测试场景合并成一个复杂大剧本。
- 不在主屏展示完整接口日志、数据库行级详情或 Prometheus 指标。
- 不替代 `/test/history` 和 `/test/report/:id` 的审计与排障价值。
- 不修改业务服务的核心交易逻辑；本设计只改变演示层与测试平台编排入口。

## 4. 页面边界

### 4.1 `/test/screen`：演示剧场

定位：评委/观众主屏。

职责：

- 一键启动标准 `UserJourney` 演示剧本。
- 自动连接 WebSocket 进度流。
- 将测试步骤映射为业务事件。
- 实时播放直播竞拍过程。
- 展示最终验收结论。
- 提供报告下钻入口。

### 4.2 `/test/user-journey`：业务验收控制台

定位：产品/开发验收入口。

职责保持不变：

- 配置 `include_reminder`、`include_sky_lamp`、`include_fixed_price`、`buyer_count` 等参数。
- 启动可配置的买家验收剧本。
- 展示步骤时间线与证据报告。

### 4.3 `/test/e2e`：竞拍专项控制台

定位：开发/技术评审入口。

职责保持不变：

- 手动配置卖家、出价者、起拍价、加价幅度、持续时间。
- 验证创建拍品、创建拍卖、出价、赢家、订单唯一等竞拍专项链路。

### 4.4 `/test/history` 与 `/test/report/:id`

定位：审计、回放、失败排障。

职责：

- `/test/history` 继续展示历史任务。
- `/test/report/:id` 继续承载详细技术证据。
- 演示剧场结束后跳转或打开报告下钻。

## 5. 核心交互

### 5.1 初始态

`/test/screen` 打开后展示：

- 标题：`AI 直播竞拍全链路验收`
- 主 CTA：`开始演示`
- 左侧角色卡片：商家、买家 A、买家 C、系统挑战。
- 中间直播间主视觉：待启动状态。
- 右侧验收结论：业务闭环、并发唯一、资产一致性，初始为待验证。

### 5.2 运行态

点击 `开始演示` 后：

1. 前端调用现有 `startUserJourney`，使用标准配置：
   - `include_reminder=true`
   - `include_sky_lamp=true`
   - `include_fixed_price=true`
   - `buyer_count=1`
   - `keep_evidence=true`
   - `auction_duration_sec` 使用适合现场演示的默认值
2. 前端通过 `discoverWS(test_id)` 获取 WebSocket 地址。
3. 前端连接测试进度流。
4. 页面将 `UserJourney` 步骤映射为视觉事件。

建议映射：

| Step | 主屏业务文案 | 视觉表现 |
| --- | --- | --- |
| `prepare` | 自动创建商品、直播间和测试买家 | 角色卡片点亮，直播间从待启动转为准备中 |
| `enter_live` | 买家进入直播间 | 直播间状态切换为 `LIVE` |
| `reminder` | 买家关注提醒 | 左侧买家卡片出现提醒标记 |
| `auction_bid` | 买家发起出价 | 当前价跳动，出价事件飘入 |
| `sky_lamp` | 点天灯触发 | 红橙色高权重飘窗，领先者切换 |
| `fixed_price_purchase` | 一口价购买完成 | 库存从 `1` 变为 `0`，订单卡片出现 |
| `verify` | 汇总校验通过 | 右侧三条验收结论变为通过 |
| `cleanup` | 清理或保留证据 | 底部显示报告入口 |

### 5.3 完成态

完成后主屏展示：

- `业务闭环成立`：进房、竞拍、成交、订单链路通过。
- `并发结果唯一`：赢家唯一、订单唯一、无重复成交。
- `资产状态一致`：库存、余额、订单状态对齐。
- `查看技术报告`：链接到 `/test/report/:test_id`。

### 5.4 失败态

失败时不能只显示 `FAIL`。

主屏应展示：

- 失败发生在哪个业务阶段，如 `点天灯`、`一口价购买`、`汇总校验`。
- 一句可读失败原因。
- `查看技术报告` 链接。
- `重新演示` 按钮。

原则：评委看懂失败发生在哪个业务环节；开发通过报告定位接口和断言。

## 6. 视觉结构

采用三栏大屏布局：

- 左侧：参演角色与系统挑战。
- 中间：直播间主视觉。
- 右侧：验收结论。

中间主视觉是页面重心：

- 大号当前价。
- 当前领先者。
- 点天灯高权重提示。
- 出价事件流。
- 订单生成。
- 库存变化。

技术信息降级显示：

- `test_id` 放底部小字。
- `WS connected/running/completed` 用小状态角标。
- 步骤断言和接口细节只在报告页展示。

## 7. 数据与接口

优先复用现有前端 API：

- `startUserJourney(config)`
- `discoverWS(test_id)`
- `cancelTest(test_id)`
- `getHistory(...)`
- `getReport(test_id)` 或现有报告轮询能力

优先复用现有 WebSocket store：

- `connected`
- `testID`
- `progress`
- `step`
- `history`
- `connect`
- `disconnect`

如果现有 `UserJourneyReport` 字段不足以驱动视觉状态，允许在测试服务报告中补充面向演示的结构化字段，但必须保持向后兼容。

建议新增可选字段：

```ts
interface DemoSnapshot {
  current_price?: string;
  leader_label?: string;
  bid_count?: number;
  order_count?: number;
  stock_before?: number;
  stock_after?: number;
  highlighted_event?: 'bid' | 'sky_lamp' | 'fixed_price' | 'order' | 'verify';
}
```

该字段是演示层快照，不作为业务事实的唯一来源；最终可信结论仍来自 `UserJourneyReport` 的断言结果。

## 8. 错误处理

- 一键启动失败：显示“演示启动失败”，保留错误原因和重试按钮。
- WebSocket 断开：显示“实时连接断开，正在读取最终报告”，继续轮询报告。
- 报告轮询超时：显示“报告生成超时”，提供 `test_id` 与重试。
- 步骤失败：主屏定位业务阶段，报告页承载详细断言。
- 用户离开页面：断开 WebSocket；不自动取消已启动任务，避免中断后台测试。

## 9. 测试策略

前端单测：

- 初始态展示“开始演示”和待验证结论。
- 点击开始后调用 `startUserJourney` 标准配置。
- WebSocket step 映射为正确的业务文案和视觉状态。
- `verify` 成功后展示三条通过结论。
- 步骤失败时展示业务阶段失败和报告入口。

后端/契约测试：

- 保持现有 `UserJourney` 场景测试。
- 如新增 `DemoSnapshot`，需补充报告序列化兼容测试。
- 不允许为了演示绕过 Gateway `/api/v1` 主链路。

手工验收：

- 打开 `/test/screen`。
- 点击 `开始演示`。
- 观察直播间主视觉随步骤推进。
- 完成后确认三条结论通过，并可跳转报告。
- 失败时确认能看到失败阶段和报告入口。

## 10. 实施切分建议

后续进入 implementation plan 时，建议拆成四个任务：

1. 重构 `Screen.tsx` 为演示剧场入口，接入一键 `UserJourney` 启动。
2. 增加 `UserJourney` step 到业务事件的前端映射层。
3. 实现直播间主视觉、验收结论和失败态。
4. 补充前端单测与必要的报告兼容测试。

## 11. 决策记录

- 目标观众：评委/观众。
- 入口方案：复用 `/test/screen`，不新增 `/test/theater`。
- 默认剧本：一键启动 `UserJourney`。
- 视觉方向：直播间主视觉优先，技术证据下钻。
- 页面边界：`E2E` 和 `UserJourney` 继续保留技术控制台职责。
