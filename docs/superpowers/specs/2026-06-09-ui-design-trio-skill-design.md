## 1. 背景与动机

用户在做 UI/UX 设计决策时，反复使用同一套人工流程：让 agent 产出多版设计、适配项目内的多套 UI 主题、用浏览器可视化对比后做选择。该流程目前每次都靠口头指令重新组织，存在两个痛点：

1. **不可复用**：每次都要重述"出三版""适配两套 UI""浏览器展示"等约束。
2. **服务不可用**：上一次实践（`past-chat-6a2707bb`，一口价上架动画）中，可视化用后台进程跑本地 HTTP 服务，agent 任务结束后进程被回收，端口失活，用户看到"服务不可用"。

本 Skill 将该流程沉淀为标准能力，并从根因上消除"服务不可用"。

## 2. 目标与非目标

### 目标
- 按用户需求产出**三版差异化 UI 设计**。
- 自动识别项目内**多套 UI 主题机制**，识别结果**透明展示**；低置信或冲突时**主动询问开发者**。
- 三版设计每一版都内置**多套 UI 切换适配**。
- 通过**稳定的本地预览服务**可视化展示，并提供**零依赖单文件 HTML 兜底**，彻底规避"服务不可用"。
- 记录用户选定方案，可衔接后续实现。

### 非目标
- 不负责把选定方案落地为生产代码（交给后续 implementation plan / 其他 skill）。
- 不强制三套 UI——若项目只有单套 UI，则不强加多套切换。

## 3. 核心工作流（6 步）

### Step 1 — UI 套数识别（自动扫描 + 透明展示 + 低置信询问）【硬规则】

1. **自动扫描代码库**识别多套 UI 机制，扫描信号包括但不限于：
   - 主题切换上下文：`themeContext`、`ThemeProvider`、`data-theme` 属性切换。
   - Design tokens：`colors.css`、`tokens/` 目录、CSS 变量 `--*`。
   - 选择器约定：`:root[data-theme='dark']` / `[data-theme='light']`。
2. **必须把识别结果明确展示给用户**，至少包含：
   - 检测到的 UI 套数与各套名称（如 `dark` / `light`）。
   - 切换机制（如 `document.documentElement` 的 `data-theme` 属性 + localStorage `h5.theme`，默认 `dark`）。
   - token 来源文件路径。
   - 用途：让用户有机会纠正误判，防止 agent 识别错了而用户不知情。
3. **置信度规则**：高置信时展示识别结果后可以继续，并提示“如识别错误请纠正”；低置信或冲突时停下来主动询问开发者，不擅自假设。
4. **闸门规则**：识别结果未经展示，不得进入后续设计步骤；只有低置信/冲突场景才需要等待用户确认。
5. **区分两种空结果**：项目根本没有前端/UI 代码（纯后端、空项目）时，直接告知"未检测到现有 UI 主题，将使用通用亮/暗双主题"并继续，不必询问；检测到前端代码但无法确定主题套数/切换机制时，才走低置信询问流程。

> 当前项目基线（供参考，仍需运行时实扫确认）：2 套 UI，`dark` / `light`，经 `frontend/h5/src/store/themeContext.tsx` 用 `data-theme` 切换，token 见 `frontend/h5/src/styles/tokens/colors.css`，默认 `dark`。

### Step 2 — 差异维度推断（自动 → 降级问答）

1. **优先**从用户需求与上下文推断三版应在哪个维度拉开差异（如：视觉风格 / 布局结构 / 动效节奏 / 交互方式）。
2. 若无法从上下文推断，则**反问用户"想从哪三方面拉开差异"**，再产出三版。

### Step 3 — 产出三版设计

- 三版在 Step 2 选定的维度上形成清晰差异。
- 每一版都**内嵌项目真实 token**（来自 Step 1 的扫描结果）。
- 每一版都内置**多套 UI 切换开关**（按 Step 1 确认的套数与机制），可在预览中实时切换查看每套 UI 下的效果。
- 若项目仅单套 UI，则不加切换开关。
- 设计前必须读取真实 token / 现有 UI 样式文件，不能凭记忆或示例值生成视觉系统。
- 可结合 `frontend-design` 或 `ui-ux-pro-max` 的设计方法提升视觉质量；当这些 skill 当前可用且任务明确要求高质量视觉方案时，优先借鉴其方法，但本 Skill 的主题识别、三版对比和稳定预览规则仍是主流程。
- 每一版必须说明差异维度、适用场景、主要风险；禁止只做颜色、文案、边距级微调来凑三版。

### Step 4 — 可视化展示（受控异步服务 + 单文件 HTML 兜底）【稳定性核心】

主路径：**受控异步本地静态服务**
1. 生成唯一预览产物：`/tmp/ui-design-trio-<timestamp>/index.html`。这个 `index.html` 必须是 self-contained 单文件，包含三版设计、多套 UI 切换、CSS 和 JS。服务预览和兜底打开都使用同一个文件，禁止维护两份内容。
2. 用当前工具环境允许的**受控异步 web server 会话**启动静态服务。使用 `RunCommand` 时必须设置 `command_type=web_server` 且 `blocking=false`，保留 `command_id`，禁止用 `&`、`nohup`、`disown` 等 shell 后台化方式伪装常驻。**端口选择**：优先让 OS 自动分配空闲端口（`http.server 0`）再从输出解析实际端口；若需固定端口，从 `8080` 起被占用则递增重试。
3. **启动后验证**：服务异步启动有绑定延迟，先等待约 500ms–1s（`CheckCommandStatus` 的 `wait_ms_before_check`）再检查命令状态（优先使用 `CheckCommandStatus`），确认进程没有立即退出；若立即退出（常见于端口占用），读取输出后换端口重试。
4. **健康探活校验**：命令仍在运行后，再用 HTTP 请求探测端口，确认真正可访问（HTTP 200）后，输出经校验的预览 URL。
5. 若当前环境有浏览器/预览工具，可打开已探活 URL；若没有，则提供已探活 URL 和同源 `index.html` 兜底路径。
6. 给出 URL 后，**不主动停止服务**，除非用户明确要求。

兜底路径：**同源零依赖单文件 HTML**
- 兜底路径就是上述唯一产物 `index.html`，不是第二份文件。
- 即使常驻服务异常失活，用户可直接用浏览器打开同一个 `index.html` 查看，**绝不失活**。
- 服务正常时不打扰用户；服务异常时立即交付同源 `index.html` 路径。

### Step 5 — 收集用户选择

- 用户选定某一版后，agent 记录决策（方案编号 + 选定理由 + 适配的 UI 套数）。

### Step 6 — 沉淀决策

- 将选定方案与 token 适配写回 `docs/superpowers/specs/`，可衔接后续 implementation plan。

## 4. 技术决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| 载体形态 | Skill | 可被主对话直接触发，也可被 subagent 复用，最贴合 TRAE 体系。 |
| 可视化主路径 | 受控异步 web server 会话 | 符合当前工具约束，agent 能继续探活和汇报，避免 shell 后台进程被静默回收。 |
| 服务实现 | `python3 -m http.server` | 零额外依赖、启动最快、最不易出问题。 |
| 兜底产物 | 同源零依赖 `index.html` | 服务预览和兜底打开使用同一文件，避免双份产物不一致；服务失活时仍可双击查看。 |
| UI 识别 | 自动扫描 + 透明展示 + 低置信询问 | 防误判、防开发者不知情；扫描失败或冲突时有询问兜底。 |
| 三版差异 | 上下文推断 + 降级问答 | 智能优先，推不出再问，避免无谓交互。 |

## 5. "服务不可用"根因与对策

| 根因 | 对策 |
|---|---|
| 后台进程被 agent 运行环境回收 | 改用受控异步 web server 会话，禁止 `&`/`nohup`/`disown` 等 shell 后台化 |
| 给出 URL 前未验证端口可访问 | 启动后强制 HTTP 健康探活，200 才给链接 |
| 双份预览内容不一致 | 只生成唯一 self-contained `index.html`；服务和兜底同源 |
| 单点依赖进程存活 | 保留同源 `index.html` 兜底，进程挂了也能看 |

`SKILL.md` 关键章节：
- **Frontmatter**: `name: ui-design-trio` + 触发性 `description`。
- **Scope**: 通用 UI 设计（不限单一仓库），但默认对当前 workspace 做 UI 扫描。
- **Mandatory Workflow**: 上述 6 步，Step 1 与 Step 4 标注为硬规则。
- **Stability Rules**: 受控异步服务 / 启动后状态检查 / 健康探活 / 同源单文件兜底 / 不主动停服。
- **Failure Handling**: 扫描失败→询问；探活失败→重启一次再降级到单文件 HTML。

## 7. 验收标准

1. 触发后能自动扫描并**展示** UI 套数识别结果，且能在扫描失败时询问开发者。
2. 产出三版**有清晰差异**的设计，每版含多套 UI 切换开关、使用真实 token，并说明差异维度、适用场景和主要风险。
3. 给出预览 URL 前完成启动后状态检查与健康探活；URL 可正常访问。
4. 即便常驻服务失活，同源 `index.html` 兜底可直接打开查看。
5. 记录用户选定方案。
