# 通用 MCP 机会图设计

- 日期：2026-06-11
- 目标：识别适合沉淀为通用 MCP 的高频开发动作，并形成产品级机会图；经二次核实后，明确区分“立刻值得做 MCP”和“先用 Skill/Script 验证”的候选
- 输入边界：基于本项目历史会话中出现的高频前后端开发动作抽象，但输出必须优先服务跨项目复用
- 输出粒度：产品级，而非接口级；强调“该不该做成 MCP”与“为什么”
- 评分方式：综合评分，维度为频率、耗时、易错性、可标准化程度、跨项目复用性、现有能力覆盖度

---

## 1. 问题定义

当前仓库已经沉淀出大量有效的 Skill、Runbook、脚本与知识库节点，但这些资产承担的职责并不相同：

- **Skill** 更擅长“理解意图、做决策、组织步骤”
- **Script/Command** 更擅长“执行某条已知路径”
- **Knowledge** 更擅长“提供背景约束与模块知识”
- **MCP** 更适合“把高价值、强结构化、可复用的能力暴露成稳定工具接口”

因此，本设计不是再总结一套方法论，也不是把所有高频动作都包装成 MCP，而是回答一个更根的问题：

> 哪些开发动作已经满足“稳定输入输出 + 高频复用 + 易错且值得标准化”，因此应该从 skill/script 层上移为 MCP 能力？

---

## 2. 设计目标与非目标

### 2.1 目标

1. 给出一张**跨项目可复用**的 MCP 机会图，而不是只服务本仓库的定制清单。
2. 区分哪些动作应该继续保留为 `skill / script / runbook`，哪些动作值得升级为 `MCP`。
3. 对每个 MCP 候选给出**产品级描述**：
   - 目标用户
   - 典型场景
   - 核心能力
   - 为什么应该做成 MCP，而不是 Skill
   - 预期收益
   - 主要风险
   - 优先级
4. 给出 `P0 / P1 / P2` 分层，便于后续决定产品化顺序。

### 2.2 非目标

- 不在本设计中下钻到具体接口 schema、tool 参数和返回结构。
- 不直接产出某个 MCP server 的实现代码。
- 不讨论“完整开发操作系统”或 workflow 总图。
- 不把纯脑力判断型任务误包装成 MCP，例如需求拆解、方案评审、代码审查结论生成。

---

## 3. MCP 判定原则

### 3.1 MCP 的第一性定义

一个动作适合做成 MCP，通常同时满足以下条件：

1. **输入结构相对稳定**  
   不是完全开放式自然语言，而是可以收敛为参数化调用。

2. **输出是“事实”或“受控动作结果”**  
   例如环境状态、契约 diff、部署验证结果、trace 聚合结果，而不是开放式主观结论。

3. **可被多个 Skill / Agent / 工作流复用**  
   不是某个单一流程里的临时步骤。

4. **人工执行成本高或易错**  
   当前通常依赖 grep、curl、shell、人工对比、反复核对。

5. **标准化收益显著**  
   一旦做成统一工具接口，可明显降低 token 消耗、重复劳动和环境差异导致的翻车概率。

### 3.2 不应做 MCP 的典型动作

以下动作即便高频，也通常**不应优先 MCP 化**：

- 强依赖开放式推理的动作：如架构设计、方案取舍、代码 review 结论
- 强依赖仓库局部上下文、难形成稳定 I/O 的动作
- 只在单一项目特殊结构下成立的动作
- 一个 shell 脚本已经足够且几乎没有复用扩展空间的动作

## 4. 评分框架

每个候选机会按 6 个维度打分，单项 1-5 分：

| 维度 | 含义 | 高分信号 |
|---|---|---|
| 频率 | 在全栈开发中出现的频次 | 几乎每轮需求、联调、发布、排障都要用 |
| 耗时 | 当前执行所需搜索、拼装、核对成本 | 依赖多轮 grep/curl/log 查询/人工对比 |
| 易错性 | 手工执行时的错误概率 | 参数多、顺序敏感、环境依赖强 |
| 可标准化程度 | 是否可收敛为稳定输入输出 | 能做成“给定参数 -> 返回结构化事实/结果” |
| 跨项目复用性 | 是否不依赖单仓库结构 | 多数微服务/全栈项目都成立 |
| 现有能力覆盖度 | 当前环境是否已有 Skill/MCP/CLI 覆盖 | 覆盖度越低，越值得新建；覆盖度高则优先复用或做编排 |

综合判断规则：

- **P0 强候选**：高频 + 高易错 + 高标准化 + 高复用，但仍需结合现有能力覆盖度判断是否立刻 MCP 化
- **P1**：价值明确，但依赖特定组织形态、技术栈或工程成熟度
- **P2**：表面高频，但更适合继续留在 Skill / Script / Runbook

二次核实规则：

- 如果已有 MCP/Skill 已覆盖 70% 以上能力，不新建 MCP，优先做编排 Skill 或补齐现有工具。
- 如果动作本质是“解释、归因、取舍”，不做 MCP，保留给 Skill。
- 如果动作本质是“结构化取事实、稳定校验、受控执行”，且现有能力覆盖不足，才进入 MCP 候选。
- 如果一个能力能用 Skill/Script 低成本验证，应先跑出复用证据，再承担 MCP 开发维护成本。

---

## 5. 机会图总览

### 5.1 P0：优先验证的强候选

1. **环境事实查询能力**：当前最有价值的空白，但不建议立刻直接做 MCP；推荐先用 Skill + Script 固化并验证复用频率，达标后再 MCP 化。

### 5.2 P1：条件成立时值得做的 MCP

1. **测试数据与演示数据编排 MCP**
2. **部署计划与发布验证 MCP**
3. **代码变更影响面分析 MCP**
4. **本地开发环境对齐 MCP**
5. **服务健康与契约验证 MCP**：已有 `api-test` 与 `post-coding-verify` 覆盖大半能力，只有在需要跨项目、非字节内网依赖的轻量契约断言时才值得独立产品化。

### 5.3 P2：不建议优先 MCP 化或已被现有能力覆盖

1. **方案规划与任务拆解**
2. **通用代码审查结论生成**
3. **知识沉淀与经验提炼**
4. **UI 方案探索与视觉对比**
5. **运行时三联诊断 MCP**：已有 `trace-query`、`metrics`、`argos-log/argos-query` 等能力，真正缺的是编排 Skill，而不是新 MCP。
6. **外部系统桥接 MCP**：GitHub 已有 MCP，飞书/Meego 等已有 CLI/Skill，优先复用现有桥接能力。

## 6. P0 强候选详解

### 6.1 环境事实查询能力

**目标用户**
- 需要快速确认“当前运行的到底是什么”的 Agent / 开发者

**典型场景**
- 本地服务与当前 worktree 是否一致
- 某个端口对应哪个进程/容器/commit
- 某个 URL 背后究竟指向哪套环境
- 浏览器里看到的页面是否来自最新代码

**核心能力**
- 读取端口、进程、容器、工作区、commit、dirty status、启动命令等环境事实
- 返回统一的“运行事实快照”
- 对比“期望代码来源”和“实际运行来源”，输出一致性判断

**为什么应该做 MCP，而不是 Skill**
- 它的核心不是单条命令查询，而是把 `端口 -> 进程 -> cwd -> worktree -> commit -> dirty status -> 目标分支` 串成稳定关联算法。
- 如果做成 MCP，返回结构化 JSON，可避免每次由 LLM 解析 `lsof`、`git status`、`docker ps` 等自由文本。
- 如果要跨 macOS、Linux、容器和远端 SSH 使用，MCP 能把平台差异封装在工具内部。

**为什么不能立刻下结论必须做 MCP**
- 这些底层命令并不复杂，Skill + 固定脚本可以先拿到 80% 效果。
- MCP 需要开发、部署和维护 server，成本高于写一段 Skill/Script。
- 当前证据只能证明这是“强候选”，不能证明它已经值得立刻产品化为 MCP。

**建议路径**
- 第一步：先做 `运行事实查询 Skill/Script`，固化快照字段和一致性判断规则。
- 第二步：在多个项目或多次排障/部署中复用，记录是否真的降低误判。
- 第三步：只有当出现“跨项目复用”或“LLM 解析 shell 文本导致误判”的证据后，再 MCP 化。

**预期收益**
- 降低“看错环境”“测错分支”“浏览器连错服务”的概率
- 让验证结论绑定明确代码来源

**主要风险**
- 不同操作系统和进程托管方式差异大
- 需要定义一套足够抽象又不过度耦合某平台的环境模型

**优先级**
- `P0 强候选，Skill/Script 先行验证`

## 7. 降级或撤销的原 P0 候选

### 7.1 服务健康与契约验证 MCP

**目标用户**
- 做联调、回归、发布验证的 Agent / 开发者

**典型场景**
- 验证某服务是否真正 ready，而不是“端口开了”
- 验证接口是否仍满足约定字段与状态码语义
- 验证某次变更是否破坏前后端契约

**核心能力**
- 对 HTTP / WS / RPC 接口执行结构化探测
- 区分“网络可达”“服务可用”“业务契约正确”
- 返回结构化验证报告，而不是一堆原始 curl 输出

**为什么应该做 MCP，而不是 Skill**
- 契约验证是典型的受控检查动作，输入输出稳定
- Skill 可以决定“该验证哪些接口”；MCP 负责执行一致的验证协议

**二次核实结论**
- 当前环境已存在 `api-test`，用于 API 测试生命周期管理。
- 当前环境已存在 `post-coding-verify`，覆盖 PPE 部署、API 测试、日志验证等后编码验证工作流。
- 因此该方向不再是 P0 空白，只能作为 P1：当需要跨项目、非特定内网平台、轻量可嵌入的契约断言工具时再考虑。

**预期收益**
- 降低把 `HTTP 200` 错看成“功能没问题”的风险
- 让发布验证、回归验证和契约回归更可复用

**主要风险**
- 不同项目的契约基线不一样，需要支持可配置的检查规则
- 如果抽象过粗，会退化成“更复杂的 curl wrapper”

**优先级**
- `P1`

---

### 7.2 运行时三联诊断 MCP

**目标用户**
- 做线上/本地问题排查的 Agent / 开发者

**典型场景**
- 一个请求为什么失败：日志、指标、trace 分别怎么看
- 某个错误是接口契约问题、性能问题还是下游依赖问题
- 想从 log/metric/trace 三个面快速聚合证据

**核心能力**
- 统一查询并关联 logs / metrics / traces
- 按 request id / user id / endpoint / time window 聚合运行时证据
- 输出结构化诊断事实，而非人工跳多个系统拼接

**为什么应该做 MCP，而不是 Skill**
- 这是典型“跨系统取证”问题，工具接口价值远大于 prompt 技巧
- Skill 可以负责诊断推理；MCP 负责高质量证据聚合

**二次核实结论**
- 当前环境已存在 `trace-query`，用于 BytedTrace 分布式追踪查询。
- 当前环境已存在 `metrics`，用于吞吐、延迟、错误率、成功率、上下游依赖和自定义指标查询。
- 当前环境还有 `argos-log/argos-query` 等日志查询能力。
- 因此这里不应新建 MCP。真正缺口是一个“诊断编排 Skill”：根据问题自动调用日志、指标、trace 工具并组织证据。

**预期收益**
- 显著缩短排障时间
- 降低“只看日志得出错误结论”的单点盲区

**主要风险**
- 接入成本高，依赖企业内部或外部观测系统权限
- 不同公司/项目的 observability stack 差异较大

**优先级**
- `P2：不新建 MCP，优先编排现有 Skill`

---

### 7.3 外部系统桥接 MCP

**目标用户**
- 需要与 GitHub、飞书、任务系统、文档系统、CI/CD 系统联动的 Agent / 开发者

**典型场景**
- 读取/更新 issue、PR、文档、白板、工单
- 将验证结果、设计结论、发布状态同步到外部系统
- 把“代码世界”的结构化事实写回“协作世界”

**核心能力**
- 将常用外部系统操作统一为结构化工具调用
- 屏蔽鉴权、对象模型、分页、格式转换等细节

**为什么应该做 MCP，而不是 Skill**
- 外部系统桥接天然是“工具问题”，不是“推理问题”
- Skill 可以决定写什么；MCP 负责稳定执行读写

**二次核实结论**
- 当前环境已存在 `mcp_GitHub`，GitHub 读写已经是 MCP。
- 当前环境已存在 `feishu-cli-read/write/import/board`、`lark-cli`、`meego-*` 等协作系统 Skill/CLI。
- 因此“通用外部系统桥接 MCP”不是空白。后续应优先复用现有 MCP/Skill，而不是重做一个宽泛桥接层。

**预期收益**
- 降低文档同步、PR 注释、状态回贴的摩擦成本
- 让 agent 真正打通“开发 -> 协作 -> 交付”链路

**主要风险**
- 权限与鉴权复杂
- 不同系统的数据模型差异大，需要谨慎做能力边界

**优先级**
- `P2：已被现有能力高度覆盖`

## 8. P1 机会详解

### 8.1 测试数据与演示数据编排 MCP

**目标用户**
- 需要稳定准备 demo 数据、测试账号、联调夹具的团队

**典型场景**
- 一键准备指定角色账号、商品、订单、竞拍状态
- 重建某个演示链路所需的数据前置条件

**核心能力**
- 参数化生成/校验/回收测试数据
- 输出数据准备状态与关键 ID

**为什么是 P1**
- 在 demo 密集型、复杂角色型项目里价值很高
- 但不是所有项目都需要“数据编排层”的 MCP

---

### 8.2 部署计划与发布验证 MCP

**目标用户**
- 需要执行或验证本地/测试/生产部署的团队

**典型场景**
- 基于当前 git/ref 生成发布 plan
- 对目标环境做只读预检查
- 发布后做统一 verify

**核心能力**
- 结构化预检查、计划生成、验证报告

**为什么是 P1**
- 通用性存在，但环境差异、发布策略、权限模型差异也很大
- 更容易先在项目内用 skill + script 跑通，再视复用度 MCP 化

---

### 8.3 代码变更影响面分析 MCP

**目标用户**
- 需要快速判断“这个改动会波及哪里”的 Agent / 开发者

**典型场景**
- 某个 API 字段变更后，前端/测试/文档哪里受影响
- 某个服务接口删除后，有哪些调用方

**核心能力**
- 基于代码索引、依赖、调用点、配置引用做结构化影响分析

**为什么是 P1**
- 很有价值，但不同语言、框架、生成代码体系差异大
- 如果基础索引能力不够强，结果容易不稳定

---

### 8.4 本地开发环境对齐 MCP

**目标用户**
- 经常遇到端口冲突、旧进程、旧容器、错误 worktree 的开发者

**典型场景**
- 确认“本地这套服务是不是干净的、唯一的、来自当前代码”
- 清理冲突资源并返回可验证状态

**核心能力**
- 检查端口、容器、进程、工作区一致性
- 输出冲突清单和可执行建议

**为什么是 P1**
- 在多 worktree、多服务本地开发中很有价值
- 但对不同 OS、进程托管方式、容器策略的依赖强

---

## 9. Skill 与 MCP 的边界判据

### 9.1 什么时候做 Skill

优先做 Skill 的情况：

- 动作本质是理解、取舍、解释、规划、归因。
- 底层工具已经存在，只缺调用顺序和判断逻辑。
- 还没有证明跨项目高频复用。
- 一段 prompt + 少量脚本就能拿到大部分收益。

例如：

- 运行时三联诊断的“归因过程”应做 Skill。
- 外部系统同步的“写什么内容”应做 Skill。
- 环境事实查询在未被验证前，应先做 Skill/Script。

### 9.2 什么时候做 MCP

优先做 MCP 的情况：

- 动作有稳定输入和稳定输出。
- 输出是事实、校验结果或受控动作结果，而不是主观结论。
- 需要被多个 Skill/Agent 调用。
- 手工解析自由文本容易出错。
- 跨 OS、跨平台、跨环境差异需要被封装。

环境事实查询如果后续 MCP 化，真正值得 MCP 化的不是 `lsof` 或 `git status` 本身，而是以下确定性关联：

```text
port/url/process -> cwd -> worktree -> branch -> commit -> dirty status -> target ref consistency
```

### 9.3 当前建议

`环境事实查询` 的当前状态是：

- **不是**“现在就必须做 MCP”
- **是**“唯一值得保留的通用 MCP 强候选”
- 最合理路径是 **Skill/Script 先行，复用证据达标后再 MCP 化**

达标信号：

- 在多个项目中复用，而不是只服务本仓库。
- 至少两类调用者依赖它，例如部署验证和排障诊断。
- LLM 解析 shell 输出出现过误判，且结构化工具能避免。
- 输出 schema 稳定，字段变化少。

---

## 10. P2：不建议优先 MCP 化的机会

### 10.1 方案规划与任务拆解

原因：
- 这本质是推理与取舍问题
- 更适合 `brainstorming`、`writing-plans`、`sdd-run` 这类 Skill 负责

### 10.2 通用代码审查结论生成

原因：
- 结论生成强依赖语义理解与风险判断
- 工具可以提供 diff、静态事实，但“是否有问题”仍主要是推理工作

### 10.3 知识沉淀与经验提炼

原因：
- 输入通常是开放式历史会话、设计文档、提交上下文
- 适合作为 knowledge/update skill，而不是 MCP

### 10.4 UI 方案探索与视觉对比

原因：
- 高度依赖创意与主观评审
- 更适合 `ui-design-trio` 一类 Skill，而不是工具接口

---

## 11. 关键边界：MCP、Skill、Script 的职责分工

推荐采用以下边界：

### 11.1 MCP

负责：
- 结构化取事实
- 执行受控动作
- 返回稳定结果

典型关键词：
- query
- verify
- inspect
- diff
- bridge
- probe

### 11.2 Skill

负责：
- 理解意图
- 决定调用顺序
- 解释事实
- 组织工作流

典型关键词：
- plan
- review
- decide
- orchestrate
- handoff

### 11.3 Script / Command

负责：
- 在单项目内执行确定性路径
- 承载环境细节与命令实现

典型关键词：
- restart
- deploy
- seed
- bootstrap

---

## 12. 推荐落地顺序

如果后续要从这张机会图继续走到实施，推荐顺序如下：

1. **先做 环境事实查询 Skill/Script**
   - 先验证快照字段、一致性判断和跨项目复用频率

2. **复用现有 api-test / post-coding-verify**
   - 不新建服务健康 MCP，除非发现跨项目轻量契约断言缺口

3. **复用现有 mcp_GitHub / feishu-cli / lark-cli**
   - 不新建泛化外部系统桥接 MCP

4. **为运行时三联诊断做编排 Skill**
   - 复用 `trace-query`、`metrics`、`argos-log`，不新建 MCP

## 13. 验收标准

本设计文档完成时，应满足：

1. 明确说明了什么是“适合 MCP 化”的动作。
2. 给出了统一评分框架，而不是拍脑袋列清单。
3. 形成经二次核实后的 `P0 强候选 / P1 条件候选 / P2 撤销或覆盖` 分层。
4. 明确说明了原四个 P0 中哪些被现有能力覆盖。
5. 明确区分了 `MCP / Skill / Script` 的职责边界。
6. 对 `环境事实查询` 给出 Skill/Script 先行、达标后 MCP 化的路径。

## 14. 一句话结论

二次核实后，原先的四个 P0 里只有 `环境事实查询` 仍然站得住，但它也不应直接 MCP 化；正确路径是先用 Skill/Script 验证“运行事实快照 + 一致性判断”的复用价值，等跨项目复用和结构化收益被证明后，再把它沉淀成 MCP。
