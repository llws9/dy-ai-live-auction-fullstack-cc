### 段 2 ⭐ 旗舰深度：出价并发三层控制（1:25–2:10）

> 深度 25% 的旗舰论据。强调「不是上一把锁这么简单」。

- **【画面】** 亮点站「出价并发：三层控制」章节，三层卡片（Redis 锁 / DB 乐观锁 / 单事务）。可穿插 `service/bid.go` 真实代码片段。
- **【操作】** 依次高亮三层卡片。
- **【口播】**
  > 「高并发出价，很多人第一反应是『上一把 Redis 锁』。但单点加锁解决不了所有问题，所以我把它拆成三层：
  > **第一层，Redis 分布式锁**——负责互斥和短时限流，锁值用纳秒时间戳防误删，解锁和续期走 Lua 脚本保证原子。
  > **第二层，数据库乐观锁**——价格更新走 `UPDATE WHERE version=?` 的 CAS 路径，冲突就重读重试，防止覆盖。
  > **第三层，单事务一致性**——出价写入和状态更新在同一个事务里，进出临界区各做一次最低价校验。
  > 我的核心取舍是：**把延时判定、通知发送、点天灯自动出价这些高成本逻辑全部移到锁外异步执行**。锁只负责保护共享状态，而不是串行化整个业务流程。」

### 段 3 实时链路：WebSocket（2:10–2:45）

> 精华版可砍。完整版保留以撑深度。

- **【画面】** 亮点站「WebSocket：完整的实时同步链路」章节。
- **【操作】** 依次指向 房间模型 / 状态恢复 / 高频广播 三块。
- **【口播】**
  > 「实时同步，重点不在『用了 WebSocket』，而在把它做成一条 **可连接、可恢复、可控广播** 的完整链路。
  > 房间按竞拍房、弹幕房、用户通知房三类隔离，慢客户端 **非阻塞丢弃**，不让一个慢连接拖垮整个房间。
  > 断线时前端指数退避重连，重连后用 `sync_request` 回灌当前价、领先者、结束时间——**恢复的是状态，不是重放日志**。
  > 高频广播两侧都做节流，排名 200 毫秒、出价 100 毫秒，避免风暴。」

### 段 4 六大难点纵深快切（2:45–3:45）

> 体现「考量维度」最密集的一段。快节奏，每个难点一句「问题 → 我的解法」。

- **【画面】** 亮点站「六大难点与解决方案」章节，逐个滚动定位。
- **【操作】** 每个难点停留约 8–10 秒，高亮关键词。
- **【口播】**
  > 「真正的复杂度集中在六个难点，我逐个讲清考量：
  > **难点一，状态机正确性**——5 个状态的有向图，状态转换全部收敛到 **纯判断模块**，避免条件散落在各处；金额统一用 decimal，杜绝 float 误差。
  > **难点二，防狙击延时**——刚才演示过了，原子累加、判定放锁外。
  > **难点三，外部依赖降级**——Redis 锁挂了退化到本地锁，弹幕限流故障就放行，AI 链路默认 fail-open。我的原则是：**绝不把『依赖可用』当成『业务可用』的前提**。
  > **难点四，点天灯自动代理出价**——类似 eBay 代理出价。用补偿替代长事务，锁粒度细化到每个订阅，并用显式标记阻断递归自激。
  > **难点五，真实在线 Presence**——连接数、用户数、头像展示是三个不同问题。在线人数以 Hub 内 presence 为权威，多标签页去重，只有鉴权身份才进头像列表，瞬时消息不污染弹幕历史。
  > **难点六，看直播领宝箱**——绝不做成前端倒计时奖励。观看时长、宝箱解锁、金币发放全部收敛到后端可信状态：心跳计时、每日分桶重置、唯一键幂等领取，而且 **金币和现金余额严格隔离**，前端动画无法刷币。」

### 段 5 AI 能力 · 双层（3:45–4:35）

- **【画面 1】** 真实 Admin → 商品「AI 一键文案」。
- **【操作】** 上传商品图 + 关键词 → 生成标题/描述/卖点/起拍价草稿。
- **【口播】**
  > 「AI 我分两层来用。**第一层是业务 AI**：商家后台『AI 一键文案』，输入商品图加关键词，调用火山方舟豆包多模态模型，输出标题、描述、卖点和建议起拍价。
  > 工程上我做了三件事：用 **JSON mode** 约束结构化输出保证稳定；对超时、上游错误、解析失败 **分级映射** 成不同状态码做失败兜底；并且把 LLM 能力收敛在 `shared/llm` 的 Provider 接口后面——**不为 AI 单独拆微服务，也不让模型 SDK 侵入业务层**。AI 只生成草稿，最终内容由商家确认。」

- **【画面 2】** 亮点站 / 终端展示 4 个自创 Skill 名称。
- **【操作】** 展示 `/dp-dev`、`/dp-prod`、`/sdd-run`、`/ui-design-trio`。
- **【口播】**
  > 「**第二层是我使用 AI 编码的方法论**。我把高频的人工经验沉淀成了 4 个自创 Skill：`/dp-dev` 本地强重启、`/dp-prod` 带风险评估的部署、`/sdd-run` 状态驱动开发、`/ui-design-trio` 三版 UI 方案评审。
  > 这背后是我的判断：**AI 代码贡献率不是越高越好，关键在于把人工经验固化成可追溯、可复用的标准流程**，让 AI 在我设定的工程约束里高效产出，而不是放任它生成不可控的代码。」

### 段 6 ⭐ 王牌差异化：测试平台 + 混沌剧场（4:35–5:30）

> 最强差异化。独立测试编排平台 + 一键混沌剧场，是「别人没有」的东西。

- **【画面 1】** 测试编排大屏（1920×1080 演示大屏）。
- **【操作】** 展示 7 类测试场景入口、实时 QPS/P99 指标面板。
- **【口播】**
  > 「这是我最想展示的差异化能力。我没有把测试停留在单测和本地脚本，而是 **单独做了一套可在线触发、可观测、可复盘的测试编排平台**。压测、E2E、防狙击、回调、混沌、剧本、AB——七类场景，执行器基于 goroutine 池加连接池复用，实时产出 QPS 和 P50/P95/P99 分位延迟。」

- **【画面 2】** 一键触发「混沌剧场模式」。
- **【操作】** 点击剧场模式 → 自动跑 `baseline → inject → recover`，韧性曲线 + 阶段旁白 + 锚点标签 + 指标卡逐步出现。
- **【口播】**
  > 「重点看这个——**混沌剧场模式，一键把混沌实验变成叙事**。系统自动跑基线、注入故障、恢复三段：进程内 RoundTripper 注入错误率、延迟、断连，曲线上用锚点标出 **故障注入时刻、SLA 击穿点、恢复拐点**，并量化检测延迟和恢复耗时。
  > 这里有一个关键工程约束：**故障注入完全不侵入业务服务主路径**，它是从测试平台发起的横切控制面，绝不在业务代码里写 if-chaos。这样我既能验证系统韧性，又不污染生产代码。」

成交，自动生成订单，中标和未中标结果统一进通知中心落库——用户离线也不会错过结果。从发布到成交，完整闭环跑通。

（并发三层）
高并发出价，很多人第一反应是"上一把 Redis 锁"。但单点加锁解决不了所有问题，所以我把它拆成三层：第一层 Redis 分布式锁，负责互斥和短时限流，锁值用纳秒时间戳防误删，解锁续期走 Lua 脚本保证原子；第二层数据库乐观锁，价格更新走 CAS 路径，冲突就重读重试，防止覆盖；第三层单事务一致性，出价写入和状态更新在同一事务里，进出临界区各做一次最低价校验。我的核心取舍是：把延时判定、通知发送、点天灯自动出价这些高成本逻辑全部移到锁外异步执行。锁只负责保护共享状态，而不是串行化整个业务流程。

（WebSocket）
实时同步，重点不在"用了 WebSocket"，而在把它做成一条可连接、可恢复、可控广播的完整链路。房间按竞拍房、弹幕房、用户通知房三类隔离，慢客户端非阻塞丢弃，不让一个慢连接拖垮整个房间。断线时前端指数退避重连，重连后用 sync_request 回灌当前价、领先者、结束时间——恢复的是状态，不是重放日志。高频广播两侧都做节流，避免风暴。

（六大难点）
真正的复杂度集中在六个难点，我逐个讲清考量：难点一状态机正确性，5 个状态的有向图，状态转换全部收敛到纯判断模块，金额统一用 decimal 杜绝 float 误差。难点二防狙击延时，原子累加、判定放锁外。难点三外部依赖降级，Redis 锁挂了退化到本地锁，弹幕限流故障就放行，AI 链路默认 fail-open，原则是绝不把"依赖可用"当成"业务可用"的前提。难点四点天灯自动代理出价，用补偿替代长事务，锁粒度细化到每个订阅，并用显式标记阻断递归自激。难点五真实在线 Presence，连接数、用户数、头像展示是三个不同问题，在线人数以 Hub 内 presence 为权威，多标签页去重，只有鉴权身份才进头像列表。难点六看直播领宝箱，绝不做成前端倒计时奖励，观看时长、宝箱解锁、金币发放全部收敛到后端可信状态，心跳计时、每日分桶重置、唯一键幂等领取，而且金币和现金余额严格隔离，前端动画无法刷币。

（AI·业务层）
AI 我分两层来用。第一层是业务 AI：商家后台"AI 一键文案"，输入商品图加关键词，调用火山方舟豆包多模态模型，输出标题、描述、卖点和建议起拍价。工程上我做了三件事：用 JSON mode 约束结构化输出保证稳定；对超时、上游错误、解析失败分级映射成不同状态码做失败兜底；并且把 LLM 能力收敛在 shared/llm 的 Provider 接口后面——不为 AI 单独拆微服务，也不让模型 SDK 侵入业务层。AI 只生成草稿，最终内容由商家确认。

（AI·工程方法论）
第二层是我使用 AI 编码的方法论。我把高频的人工经验沉淀成了 4 个自创 Skill：dp-dev 本地强重启、dp-prod 带风险评估的部署、sdd-run 状态驱动开发、ui-design-trio 三版 UI 方案评审。这背后是我的判断：AI 代码贡献率不是越高越好，关键在于把人工经验固化成可追溯、可复用的标准流程，让 AI 在我设定的工程约束里高效产出，而不是放任它生成不可控的代码。

（测试平台）
这是我最想展示的差异化能力。我没有把测试停留在单测和本地脚本，而是单独做了一套可在线触发、可观测、可复盘的测试编排平台。压测、E2E、防狙击、回调、混沌、剧本、AB——七类场景，执行器基于 goroutine 池加连接池复用，实时产出 QPS 和 P50/P95/P99 分位延迟。

（混沌剧场）
重点看这个——混沌剧场模式，一键把混沌实验变成叙事。系统自动跑基线、注入故障、恢复三段：进程内 RoundTripper 注入错误率、延迟、断连，曲线上用锚点标出故障注入时刻、SLA 击穿点、恢复拐点，并量化检测延迟和恢复耗时。这里有一个关键工程约束：故障注入完全不侵入业务服务主路径，它是从测试平台发起的横切控制面，绝不在业务代码里写 if-chaos。

（可观测性）
可观测性，我的理解不是"出事了再看日志"，而是从请求、业务指标到实验效果有统一观测入口。结构化 JSON 日志用 X-Request-ID 跨服务串联，敏感字段统一脱敏进 Loki。业务指标覆盖出价、延时触发、并发峰值、溢价率、GMV、支付成功率。更进一步，我把增长漏斗也做成了可观测链路：提醒点击、进房回流、点击出价、一口价点击、购买成功，全程可上报、可落库、可聚合。

（收尾）
最后总结。这个项目我想证明的不是"功能做完了"，而是工程深度：并发用三层而非单点、实时链路可恢复、外部依赖会降级、测试能复盘、AI 可控可追溯。贯穿始终的是一句话——绝不把"依赖可用"当成"业务可用"的前提，每一处都做到失败关闭、优雅降级。课题基线要的是两端闭环，我交付的是一套可上线、可观测、可演进的生产级系统。谢谢。
