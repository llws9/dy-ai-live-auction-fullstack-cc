### 段 2 ⭐ 旗舰深度：出价并发三层控制（1:25–2:10）

> 深度 25% 的旗舰论据。强调「不是上一把锁这么简单」。

- **【画面】** 亮点站「出价并发：三层控制」章节，三层卡片（Redis 锁 / DB 乐观锁 / 单事务）。可穿插 `service/bid.go` 真实代码片段。
- **【操作】** 依次高亮三层卡片。
- **【口播】**
  > 「高并发出价，可能很多人第一反应是上一把 Redis 锁就够了。但单点加锁其实解决不了所有问题，所以我把它拆成了三层。
  > **第一层是 Redis 分布式锁**，负责互斥和短时限流，锁值用纳秒时间戳防止误删，解锁和续期都走 Lua 脚本保证原子。
  > **第二层是数据库乐观锁**，价格更新走 `UPDATE WHERE version=?` 的 CAS 路径，遇到冲突就重读重试，防止被覆盖。
  > **第三层是单事务一致性**，出价写入和状态更新放在同一个事务里，进出临界区各做一次最低价校验。
  > 我自己觉得最关键的一个取舍是：**把延时判定、通知发送、点天灯自动出价这些比较重的逻辑全都挪到锁外面异步做**，让锁只负责保护共享状态，而不是把整个业务流程都串行化。」

> 精华版可砍。完整版保留以撑深度。

- **【画面】** 亮点站「WebSocket：完整的实时同步链路」章节。
- **【操作】** 依次指向 房间模型 / 状态恢复 / 高频广播 三块。
- **【口播】**
  > 「实时同步这块，我觉得重点不只是用了 WebSocket，而是要把它做成一条 **能连上、能恢复、广播也可控** 的完整链路。
  > 房间我按竞拍房、弹幕房、用户通知房三类分开管理，遇到慢客户端就 **非阻塞丢弃**，不让一个慢连接拖垮整个房间。
  > 断线的时候前端会指数退避重连，重连之后用 `sync_request` 把当前价、领先者、结束时间重新拉回来——**恢复的是状态，而不是把日志重放一遍**。
  > 高频广播两边都做了节流，避免消息风暴。」

### 段 4 六大难点纵深快切（2:45–3:45）

> 体现「考量维度」最密集的一段。快节奏，每个难点一句「问题 → 我的解法」。

- **【画面】** 亮点站「六大难点与解决方案」章节，逐个滚动定位。
- **【操作】** 每个难点停留约 8–10 秒，高亮关键词。
- **【口播】**
  > 「系统真正复杂的地方集中在六个难点，我一个个说说我的考量：
  > **难点一是状态机的正确性**，我把 5 个状态画成有向图，状态转换全部收敛到 **纯判断模块** 里，避免条件散落在各个地方；金额也统一用 decimal，避免 float 比较出误差。
  > **难点二是防狙击延时**，刚才演示过了，就是原子累加加上判定放锁外。
  > **难点三是外部依赖降级**，Redis 锁挂了就退回本地锁，弹幕限流出问题就放行，AI 链路默认 fail-open。我的原则是 **绝对不能把『依赖可用』当成『业务可用』的前提**。
  > **难点四是点天灯，也就是自动代理出价**，有点像 eBay 的代理出价。我用补偿代替长事务，把锁粒度细化到每个订阅，再用显式标记挡住递归自激。
  > **难点五是真实在线 Presence**，连接数、用户数、头像展示其实是三个不同的问题。在线人数以 Hub 里的 presence 为准，多标签页去重，只有鉴权过的身份才能进头像列表，瞬时消息也不会污染弹幕历史。
  > **难点六是看直播领宝箱**，我没做成前端倒计时奖励，而是把观看时长、宝箱解锁、金币发放全部收敛到后端可信状态：靠心跳计时、每日分桶重置、唯一键幂等领取，而且 **金币和现金余额是严格隔离的**，前端动画刷不了币。」

- **【画面 1】** 真实 Admin → 商品「AI 一键文案」。
- **【操作】** 上传商品图 + 关键词 → 生成标题/描述/卖点/起拍价草稿。
- **【口播】**
  > 「AI 我是分两层来用的。**第一层是业务上的 AI**：商家后台这个『AI 一键文案』，输入商品图加几个关键词，就会调用火山方舟豆包的多模态模型，帮商家生成标题、描述、卖点和建议起拍价。
  > 工程上我主要做了三件事：用 **JSON mode** 约束结构化输出让它更稳定；把超时、上游错误、解析失败 **分级映射** 成不同状态码做兜底；还有把 LLM 能力收敛在 `shared/llm` 的 Provider 接口后面——**没有为 AI 单独拆一个微服务，也没让模型 SDK 侵入业务层**。AI 在这里只生成草稿，最终内容还是商家自己确认。」

- **【画面 2】** 亮点站 / 终端展示 4 个自创 Skill 名称。
- **【操作】** 展示 `/dp-dev`、`/dp-prod`、`/sdd-run`、`/ui-design-trio`。
- **【口播】**
  > 「**第二层是我自己用 AI 写代码的方法论**。我把平时高频用到的人工经验沉淀成了 4 个自创 Skill：`/dp-dev` 本地强重启、`/dp-prod` 带风险评估的部署、`/sdd-run` 状态驱动开发、`/ui-design-trio` 三版 UI 方案评审。
  > 我的想法是，**AI 代码贡献率其实不是越高越好，关键是把人工经验固化成可追溯、可复用的标准流程**，让 AI 在我设定好的工程约束里去高效产出，而不是放任它生成一堆不可控的代码。」

### 段 6 ⭐ 王牌差异化：测试平台 + 混沌剧场（4:35–5:30）

> 最强差异化。独立测试编排平台 + 一键混沌剧场，是「别人没有」的东西。

- **【画面 1】** 测试编排大屏（1920×1080 演示大屏）。
- **【操作】** 展示 7 类测试场景入口、实时 QPS/P99 指标面板。
- **【口播】**
  > 「接下来是我自己挺想给大家看的一块。我没有把测试只停留在单测和本地脚本，而是 **单独做了一套可以在线触发、可观测、还能复盘的测试编排平台**。压测、E2E、防狙击、回调、混沌、剧本、AB，一共七类场景，执行器是基于 goroutine 池加连接池复用做的，能实时算出 QPS 和 P50/P95/P99 这些分位延迟。」

- **【画面 2】** 一键触发「混沌剧场模式」。
- **【操作】** 点击剧场模式 → 自动跑 `baseline → inject → recover`，韧性曲线 + 阶段旁白 + 锚点标签 + 指标卡逐步出现。
- **【口播】**
  > 「重点看这个 **混沌剧场模式，它能一键把一次混沌实验变成一段叙事**。系统会自动跑基线、注入故障、恢复这三段，用进程内的 RoundTripper 去注入错误率、延迟和断连，曲线上还会用锚点标出 **故障注入的时刻、SLA 击穿的点和恢复的拐点**，并且把检测延迟和恢复耗时都量化出来。
  > 这里有个我比较在意的工程约束：**故障注入完全不碰业务服务的主路径**，它是从测试平台发起的横切控制面，绝对不会在业务代码里写 if-chaos。这样既能验证韧性，又不会污染生产代码。」

用户进来直播间后，大家可以注意一下右上角，这里的在线头像和人数是真实在线的 Presence 系统，同一个人开多个标签页也只算一个人，而不是简单数 WebSocket 连接数。

（A 出价）
用户 A 出价后，会有领先状态胶囊、游戏化出价飘屏、战况热度条，让用户在关键的 5 秒里一眼看懂"我是否领先、要不要反超、这场热不热"。

（B 反超）
接着用户 B 出价反超了 A，A 这边会立刻收到"被超越"的推送。这条推送是出价成功后实时写进去、再通过用户专属的 WebSocket 房间精准推给他的，不是广播。

（防狙击延时）
再看这个挺关键的防狙击。如果快结束的时候还有人出价，系统会自动延时，避免有人卡着最后一秒绝杀。延时是用 SQL 原子累加做的，而且判定放在锁外面，不会放大锁竞争。

（成交）
成交之后会自动生成订单，中标和没中标的结果都会进通知中心存下来，用户就算当时离线了也不会错过。这样从发布到成交，整条链路就完整跑通了。

（并发三层）
高并发出价，可能很多人第一反应是上一把 Redis 锁就够了。但单点加锁其实解决不了所有问题，所以我把它拆成了三层。第一层是 Redis 分布式锁，负责互斥和短时限流，锁值用纳秒时间戳防止误删，解锁和续期都走 Lua 脚本保证原子。第二层是数据库乐观锁，价格更新走 CAS 路径，遇到冲突就重读重试，防止被覆盖。第三层是单事务一致性，出价写入和状态更新放在同一个事务里，进出临界区各做一次最低价校验。我自己觉得最关键的一个取舍是，把延时判定、通知发送、点天灯自动出价这些比较重的逻辑全都挪到锁外面异步做，让锁只负责保护共享状态，而不是把整个业务流程都串行化。

（WebSocket）
实时同步这块，我觉得重点不只是用了 WebSocket，而是要把它做成一条能连上、能恢复、广播也可控的完整链路。房间我按竞拍房、弹幕房、用户通知房三类分开管理，遇到慢客户端就非阻塞丢弃，不让一个慢连接拖垮整个房间。断线的时候前端会指数退避重连，重连之后用 sync_request 把当前价、领先者、结束时间重新拉回来，恢复的是状态，而不是把日志重放一遍。高频广播两边都做了节流，避免消息风暴。

（六大难点）
系统真正复杂的地方集中在六个难点，我一个个说说我的考量。难点一是状态机的正确性，我把 5 个状态画成有向图，状态转换全部收敛到纯判断模块里，避免条件散落在各个地方，金额也统一用 decimal，避免 float 比较出误差。难点二是防狙击延时，刚才演示过了，就是原子累加加上判定放锁外。难点三是外部依赖降级，Redis 锁挂了就退回本地锁，弹幕限流出问题就放行，AI 链路默认 fail-open，我的原则是绝对不能把"依赖可用"当成"业务可用"的前提。难点四是点天灯，也就是自动代理出价，有点像 eBay 的代理出价，我用补偿代替长事务，把锁粒度细化到每个订阅，再用显式标记挡住递归自激。难点五是真实在线 Presence，连接数、用户数、头像展示其实是三个不同的问题，在线人数以 Hub 里的 presence 为准，多标签页去重，只有鉴权过的身份才能进头像列表。难点六是看直播领宝箱，我没做成前端倒计时奖励，而是把观看时长、宝箱解锁、金币发放全部收敛到后端可信状态，靠心跳计时、每日分桶重置、唯一键幂等领取，而且金币和现金余额是严格隔离的，前端动画刷不了币。

（AI·业务层）
AI 我是分两层来用的。第一层是业务上的 AI，商家后台这个"AI 一键文案"，输入商品图加几个关键词，就会调用火山方舟豆包的多模态模型，帮商家生成标题、描述、卖点和建议起拍价。工程上我主要做了三件事，一是用 JSON mode 约束结构化输出让它更稳定，二是把超时、上游错误、解析失败分级映射成不同状态码做兜底，三是把 LLM 能力收敛在 shared/llm 的 Provider 接口后面，没有为 AI 单独拆一个微服务，也没让模型 SDK 侵入业务层。AI 在这里只生成草稿，最终内容还是商家自己确认。

（AI·工程方法论）
第二层是我自己用 AI 写代码的方法论。我把平时高频用到的人工经验沉淀成了 4 个自创 Skill，dp-dev 是本地强重启，dp-prod 是带风险评估的部署，sdd-run 是状态驱动开发，ui-design-trio 是三版 UI 方案评审。我的想法是，AI 代码贡献率其实不是越高越好，关键是把人工经验固化成可追溯、可复用的标准流程，让 AI 在我设定好的工程约束里去高效产出，而不是放任它生成一堆不可控的代码。

（测试平台）
接下来是我自己挺想给大家看的一块。我没有把测试只停留在单测和本地脚本，而是单独做了一套可以在线触发、可观测、还能复盘的测试编排平台。压测、E2E、防狙击、回调、混沌、剧本、AB，一共七类场景，执行器是基于 goroutine 池加连接池复用做的，能实时算出 QPS 和 P50/P95/P99 这些分位延迟。

（混沌剧场）
重点看这个混沌剧场模式，它能一键把一次混沌实验变成一段叙事。系统会自动跑基线、注入故障、恢复这三段，用进程内的 RoundTripper 去注入错误率、延迟和断连，曲线上还会用锚点标出故障注入的时刻、SLA 击穿的点和恢复的拐点，并且把检测延迟和恢复耗时都量化出来。这里有个我比较在意的工程约束，就是故障注入完全不碰业务服务的主路径，它是从测试平台发起的横切控制面，绝对不会在业务代码里写 if-chaos，这样既能验证韧性，又不会污染生产代码。

（可观测性）
可观测性这块，我的理解不是出了事再回头翻日志，而是从请求、业务指标到实验效果都有一个统一的入口。结构化的 JSON 日志用 X-Request-ID 跨服务串起来，敏感字段统一脱敏后进 Loki。业务指标覆盖了出价、延时触发、并发峰值、溢价率、GMV、支付成功率这些。再往前一步，我把增长漏斗也做成了可观测链路，从提醒点击、进房回流、点击出价、一口价点击到购买成功，全程都能上报、落库、聚合。

（收尾）
最后简单总结一下。这个项目我其实想说明的不是功能做完了，而是它有工程深度，并发用了三层而不是单点，实时链路能恢复，外部依赖会降级，测试能复盘，AI 也可控可追溯。这里面贯穿始终的就一句话，绝不把"依赖可用"当成"业务可用"的前提，每个地方都尽量做到失败关闭、优雅降级。课题要的是两端闭环，而我想交付的，是一套能上线、能观测、也能继续演进的系统。谢谢大家。
