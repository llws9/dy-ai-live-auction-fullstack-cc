# Admin 双角色可见性设计方案

## 1. 背景

当前 `frontend/admin` 已具备管理端页面、登录态保护和基础角色字段展示，但尚未形成页面级、菜单级和数据级的角色权限模型。现状是：

- 路由统一由登录态保护，未按 `user.role` 做页面访问控制。
- 菜单配置为静态结构，未按角色动态过滤。
- `403` 仅作为接口错误提示处理，未形成统一的未授权页面或路由处理。
- 角色语义已存在：`1` 表示商家/主播，`2` 表示管理员。

本方案用于定义平台管理员与商家在管理端前端中的页面可见性、操作边界和后端兜底原则。

## 2. 目标

- 明确平台管理员和商家分别应该看到哪些页面。
- 明确同一页面在不同角色下的数据范围差异。
- 明确平台管理员不具备代运营能力，不允许代商家创建、编辑或上下架经营资产。
- 为后续菜单过滤、路由拦截、API 数据范围控制提供统一依据。

## 3. 非目标

- 不设计完整 RBAC 权限系统。
- 不引入平台管理员代运营能力。
- 不拆分为两个独立前端应用。
- 不在本方案中定义后端表结构或审计日志表结构。
- 不改变当前角色枚举语义。

## 4. 角色定义

### 4.1 平台管理员

平台管理员负责平台治理，不负责商家经营。核心职责包括：

- 查看全平台经营、竞拍、直播间、订单和统计数据。
- 查看商品、竞拍、直播间等经营资产，用于治理、排查和风控。
- 管理平台角色与用户。
- 查看平台级用户统计。

平台管理员不具备以下能力：

- 不创建商品。
- 不编辑商家商品。
- 不创建直播间。
- 不配置商家竞拍规则模板。
- 不执行一口价上下架。
- 不代商家完成任何经营动作。

### 4.2 商家

商家负责自身经营闭环。核心职责包括：

- 管理自己的商品。
- 管理自己的直播间。
- 管理自己的竞拍活动。
- 管理自己的竞拍参数模板。
- 管理自己直播间内的一口价商品上下架。
- 查看自己的订单和经营统计。

商家不具备以下能力：

- 不查看跨商家数据。
- 不访问平台用户统计。
- 不管理平台角色。
- 不管理平台用户。

## 5. 设计原则

- 前端体验原则：未授权页面不出现在菜单中。
- 前端安全原则：直接输入 URL 也必须被路由层拦截。
- 后端兜底原则：前端权限只改善体验，真实权限必须由后端接口按角色和数据归属校验。
- 数据范围原则：平台管理员使用 `all` 数据范围，商家使用 `owner_only` 数据范围。
- 经营资产原则：商品、直播间、竞拍规则、一口价上下架属于商家经营资产，平台管理员默认只读或不可见。
- 最小能力原则：不因为管理员身份而自动拥有所有写权限。

## 7. 菜单设计

### 7.1 平台管理员菜单

平台管理员菜单保留治理和全局查看能力：

- 经营总览：`/dashboard`
- 商品管理：`/goods/list`
- 竞拍管理：`/auction/list`
- 直播间管理：`/live/list`
- 订单管理：`/order/list`
- 数据统计：`/stats/auction`、`/stats/revenue`、`/stats/user`
- 系统设置：`/system/profile`、`/system/permission/roles`、`/system/permission/users`

平台管理员菜单不显示：

- `/goods/create`
- `/goods/edit`
- `/auction/rules`
- `/auction/rules/create`
- `/auction/rules/edit`
- `/live/create`
- `/live/fixed-price`

### 7.2 商家菜单

商家菜单保留自身经营闭环：

- 经营总览：`/dashboard`
- 商品管理：`/goods/list`、`/goods/create`
- 竞拍管理：`/auction/list`、`/auction/rules`
- 直播间管理：`/live/list`、`/live/create`、`/live/fixed-price`
- 订单管理：`/order/list`
- 数据统计：`/stats/auction`、`/stats/revenue`
- 系统设置：`/system/profile`

商家菜单不显示：

- `/stats/user`
- `/system/permission/roles`
- `/system/permission/users`

### 7.3 操作型页面

详情页、编辑页和创建页不一定出现在菜单中，但必须配置路由权限。包括：

- `/goods/edit`
- `/auction/detail`
- `/auction/rules/create`
- `/auction/rules/edit`
- `/live/detail`
- `/order/detail`

## 8. 路由权限设计

前端路由应从单一登录态保护升级为登录态与角色权限双层保护。

推荐模型：

- `RequireAuth`：只负责登录态校验。
- `RequireRole`：负责角色可见性校验。
- `routeAccessMap`：作为前端路由权限的 SSOT。
- `navItems`：从同一份权限配置派生菜单，避免菜单与路由权限不一致。

未授权访问处理：

- 未登录访问受保护页面：重定向到 `/admin-login`。
- 已登录但角色无权限访问：展示 `403` 页面，或重定向到 `/dashboard` 并提示“无权限访问该页面”。
- 推荐优先展示 `403` 页面，便于区分未登录和无权限。

## 9. 数据范围设计

页面可见不等于数据可见。同一页面在不同角色下必须使用不同数据范围：

| 角色 | 数据范围 | 含义 |
|---|---|---|
| 平台管理员 | `all` | 可查看全平台数据 |
| 商家 | `owner_only` | 只能查看当前商家主体拥有的数据 |

商家数据范围必须由后端基于 JWT 派生身份判断，不能依赖前端传入的商家 ID。

后端接口要求：

- 列表接口必须按角色过滤数据。
- 详情接口必须校验资源归属。
- 写接口必须校验资源归属和操作权限。
- 平台管理员对商家经营资产不拥有默认写权限。

## 10. 操作权限设计

| 能力 | 平台管理员 | 商家 |
|---|---:|---:|
| 查看全平台商品 | 允许 | 不允许 |
| 创建商品 | 不允许 | 允许 |
| 编辑商品 | 不允许 | 仅自己的商品 |
| 查看全平台竞拍 | 允许 | 不允许 |
| 创建/编辑竞拍规则模板 | 不允许 | 仅自己的模板 |
| 查看全平台直播间 | 允许 | 不允许 |
| 创建直播间 | 不允许 | 允许 |
| 一口价上下架 | 不允许 | 仅自己的直播间 |
| 查看全平台订单 | 允许 | 不允许 |
| 查看自身订单 | 不适用 | 允许 |
| 查看用户统计 | 允许 | 不允许 |
| 管理角色 | 允许 | 不允许 |
| 管理用户 | 允许 | 不允许 |

## 11. 前端实现边界

前端应承担以下职责：

- 根据 `user.role` 动态过滤菜单。
- 根据路由权限配置拦截未授权页面。
- 在页面内根据角色隐藏无权限操作按钮。
- 将通用页面的数据 scope 交由 API 返回结果表达，不在前端伪造跨角色数据。
- 对 `403` 返回做统一提示或跳转。

前端不应承担以下职责：

- 不通过前端传入 `owner_id` 决定商家数据范围。
- 不通过隐藏按钮作为唯一权限控制。
- 不允许平台管理员通过 URL 参数选择商家后执行经营动作。

## 12. 后端兜底要求

后端必须保证：

- 从 JWT 派生 `X-User-ID` 和 `X-User-Role`。
- 商家接口按当前用户或商家主体过滤资源。
- 管理员只在明确的平台治理接口中拥有写权限。
- 商家访问非本人资源时返回 `403` 或 `404`，具体策略由接口安全要求决定。
- 经营资产写操作不得因 `role=2` 自动放行。

## 13. 验收标准

- 管理员登录后不看到商家经营动作菜单。
- 商家登录后不看到平台治理菜单。
- 商家直接访问平台治理路由时被拦截。
- 管理员直接访问商家经营动作路由时被拦截。
- 商家列表和详情接口只能返回自身数据。
- 管理员列表和详情接口可以返回全平台数据，但不提供代运营写操作。
- `/auction/rules` 仅作为商家自定义竞拍参数模板入口。

## 14. 推荐落地顺序

1. 建立前端路由权限配置 SSOT。
2. 从权限配置派生菜单。
3. 增加 `RequireRole` 路由保护。
4. 增加统一 `403` 页面或无权限提示。
5. 页面内按角色隐藏无权限操作按钮。
6. 后端补齐数据范围与写权限兜底。
7. 添加前端路由权限测试与关键接口权限测试。
