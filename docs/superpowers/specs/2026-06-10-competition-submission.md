## 4. 核心功能清单

按「用户路径 + 系统能力」拆分，覆盖课题要求的完整闭环并向生产级延展：

1. **商家端竞拍管理（PC Admin）**：商品上架、竞拍规则配置（0 元起拍 / 加价幅度 / 封顶价 / 时长 / 自动延时）、商品与场次管理、异常竞拍取消、订单管理；并提供 **AI 一键文案**（多模态生成标题、描述、卖点、建议起拍价）。
2. **用户端实时竞拍（移动 H5）**：直播间浏览、商品/规则/当前价/在线人数查看、手动出价、实时排名、「被超越 / 延时 / 结束 / 中标」关键提醒、模拟支付、历史记录与浏览足迹。
3. **实时出价与状态机引擎**：基于 WebSocket 的毫秒级同步，出价校验、动态排名、5 状态竞拍状态机、防狙击自动延时、封顶价自动成交、成交自动生成订单。
4. **高并发出价一致性保障**：Redis 分布式锁 + 数据库乐观锁（CAS） + 单事务三层并发控制，保证「绝不一笔出价扣两次钱」的幂等性。
5. **沉浸式竞价氛围**：领先状态胶囊、游戏化出价飘屏、战况热度条（自动衰减三档）、半嵌入式战况坞，强化「领先/被超越/白热化」的情绪反馈。
6. **进阶玩法与运营能力**：点天灯（自动代理出价，类 eBay proxy bidding）、真实在线 Presence（头像+观看人数）、看直播领宝箱（后端可信计时+幂等发币+资产隔离）、增长漏斗埋点、A/B 实验（GrowthBook）。
7. **独立测试编排平台 + 可观测性**：在线触发 7 类测试场景（压测/E2E/防狙击/回调/混沌/剧本/AB），混沌剧场模式一键演示「基线→注入→恢复」；Prometheus + Grafana + Loki 统一观测面。

## 5. 端到端使用流程

1. 商家登录 Admin 后台，上架商品并配置竞拍规则（起拍价、加价幅度、封顶价、时长、自动延时），可用 AI 一键生成文案后确认发布。
2. 用户打开移动端 H5 进入直播间，看到商品详情、当前价、实时在线头像与观看人数，以及战况热度条。
3. 用户点击出价，请求经网关 `/api/v1` 进入竞拍服务，通过「Redis 锁 + 乐观锁 + 单事务」三层校验后写入，最新价、排名通过 WebSocket 秒级广播给全房间。
4. 当有人反超时，被超越用户通过其专属 WebSocket 通知房间立即收到「被超越」推送（精准推送而非广播）。
5. 竞拍临近结束若仍有出价，系统触发防狙击自动延时（SQL 原子累加、判定放锁外），杜绝「最后一秒绝杀」。
6. 达到封顶价或倒计时结束，状态机流转到成交，自动生成订单；中标/未中标结果统一进通知中心落库，用户离线也能补收。
7. 用户在 H5 完成模拟支付，并可在「我的订单 / 钱包 / 浏览足迹」查看交易闭环与流水。
8. 全过程的出价、延时、并发峰值、GMV、支付成功率等指标实时进入 Grafana；评委可在测试编排平台一键回放高并发与混沌韧性实验。

## 11. 大模型 / AI 能力使用说明

AI 分两层落地，对齐评分「工具适配 + 流程可追溯 + 贡献率合理性」。

### 11.1 业务侧 AI——商家一键文案
- **模型**：火山方舟 Ark 的 **Doubao（Doubao-Seed-2.0-lite）**，通过 OpenAI 兼容的 `/chat/completions` 接口调用（实现见 `backend/shared/llm/doubao.go`）。
- **能力**：多模态输入（商品图 `image_url` + 关键词文本），输出商品标题、描述、卖点、建议起拍价。
- **位置**：封装在 `backend/shared/llm` 的 `Provider` 接口后（`provider.go`），由 Admin 商家端调用。**不为 AI 单独拆微服务，模型 SDK 不侵入业务层**。
- **工程化保障**：
  - **结构化输出**：使用 `response_format: json_object`（JSON mode）约束输出，保证可解析、可落库。
  - **失败分级兜底**：上游错误映射为 `ErrUpstreamTimeout / ErrUpstreamClient / ErrUpstreamServer / ErrInvalidResponse / ErrMissingCredentials` 五类，分别转换为不同状态码与降级策略，AI 链路默认 **fail-open**（失败不阻塞商家发布）。
  - **可观测**：每次调用打点 `request_start / response_received / request_success`，记录 endpoint、模型、token 用量、耗时。
  - **人工把控**：AI 只生成**草稿**，最终内容由商家确认后发布。

### 11.2 工程侧 AI 使用方法论——4 个自创 Skill
将高频人工经验沉淀为可追溯、可复用的标准流程，让 AI 在设定的工程约束内高效产出：
- `/dp-dev`：本地强重启，标准化本地联调环境。
- `/dp-prod`：带风险评估的部署流程（SSH/RSYNC 重试、版本一致性校验、就绪后安全 seed）。
- `/sdd-run`：状态驱动开发（SSOT 状态文件 + TDD + 子任务交付证据）。
- `/ui-design-trio`：三版 UI 方案评审，沉淀设计决策。
- **理念**：AI 代码贡献率不是越高越好，关键在于把**关键决策点**的人工把控固化为流程，而非放任生成不可控代码。

## 12. 关键工程难点与解决方案

> 六大难点图文详解见亮点站：https://llws9.github.io/auction-highlights/#difficulties

1. **高并发出价一致性（核心）**
   - 问题：100+ 用户同时狂点出价，要防超卖、防重复扣款、防价格覆盖。
   - 方案：**三层控制**。① Redis 分布式锁（纳秒时间戳防误删、Lua 脚本原子解锁/续期）做互斥与短时限流；② 数据库乐观锁 `UPDATE WHERE version=?`（CAS，冲突重读重试）防覆盖；③ 出价写入与状态更新放在单事务，进出临界区各做一次最低价校验。**核心取舍**：延时判定、通知发送、点天灯自动出价等重逻辑全部移到锁外异步，锁只保护共享状态，不串行化业务流程。

2. **毫秒级实时同步与断连恢复**
   - 问题：排名秒级一致、倒计时无偏差、网络波动自动重连。
   - 方案：WebSocket 房间级隔离（竞拍房/弹幕房/用户通知房），慢客户端非阻塞丢弃；前端指数退避重连，重连后 `sync_request` 回灌当前价/领先者/结束时间（恢复状态而非重放日志）；排名 200ms、出价 100ms 双侧节流防风暴。
   - **真实线上复盘 · 高频出价下抽屉价格滞后**：演示中连续触发并发出价后，出价抽屉显示当前价 ¥440，用户按 ¥450 出价持续失败，最终成交价却是 ¥460——抽屉读到的是过期价格。根因是抽屉 `current_price` 主要依赖 WS 增量推送，高频点击下 WS 末态延迟/漏更，据旧价算出的最低加价已低于真实最低价。修复不把一致性押在 WS 单一通道：新增 demo 竞拍同步事件 `demoAuctionEvents`，`DemoConsole` 在并发压测成功后广播后端权威 `highest_amount`，`LiveRoomSlide` 监听并对 `current_price` 做单调向上修正、即时刷新抽屉 `minBid/bidAmount`。补 2 个回归测试（`npm test` 66 通过、`build` 通过），实测抽屉报价始终不低于服务端真实最高价。详见亮点站 https://llws9.github.io/auction-highlights/#websocket

3. **防狙击自动延时**
   - 问题：杜绝「最后一秒绝杀」，又不能放大锁竞争。
   - 方案：SQL 原子累加实现延时，延时判定放在锁外执行，避免拉长临界区。

4. **状态机正确性与金额精度**
   - 问题：5 状态流转零漏洞、金额无浮点误差。
   - 方案：5 状态有向图，状态转换收敛到纯判断模块避免条件散落；金额统一用 `shopspring/decimal`，杜绝 float 误差。

5. **外部依赖降级（失败关闭）**
   - 问题：Redis / AI / 限流等依赖故障不能拖垮主流程。
   - 方案：Redis 锁不可用退化本地锁，弹幕限流故障放行，AI 链路默认 fail-open；统一原则——绝不把「依赖可用」当「业务可用」前提。

6. **进阶玩法的可信与隔离**
   - 点天灯（自动代理出价）：用补偿替代长事务，锁粒度细化到每个订阅，显式标记阻断递归自激。
   - 真实在线 Presence：在线人数以 Hub 内 presence 为权威、多标签页按 user_id 去重，仅鉴权身份进头像列表，瞬时消息不进弹幕历史。
   - 看直播领宝箱：后端可信心跳计时、每日分桶重置、唯一键幂等领取，金币与现金余额严格隔离，前端动画无法刷币。

### 14.1 性能指标 / 压测结果
- **压测 & 混沌韧性报告（可视化专题页）**：https://llws9.github.io/auction-highlights/perf.html —— 含读路径分位延迟、SLA 达标表、混沌剧场韧性曲线（基线→注入→恢复，三种注入场景可切换）。
- 真实压测数据（本地环境，Apache Bench）：读操作 QPS ~2000（目标 >500，超 4×）、API 平均响应 <1ms、竞拍详情 P99 42ms、读接口成功率 100%。
- 写路径以一致性优先：三层并发控制保证出价幂等不超卖；并发出价吞吐 ~170 QPS（受 DB 外键校验影响，已识别逻辑外键优化项）。
- 混沌剧场模式量化系统韧性：故障检测延迟、SLA 击穿点、恢复耗时。
- 设计目标对齐课题加分项：单直播间 1000+ 用户同时在线（房间级路由隔离）。
- 数据源 `docs/performance_test_report.md` 与测试大屏实时面板：http://14.103.53.55/test-dashboard/

### 14.2 Prompt 策略 / 失败兜底机制
- **Prompt 策略**：商家文案生成采用「系统约束 + 多模态商品图 + 关键词」结构，强制 `response_format=json_object` 输出结构化字段（标题/描述/卖点/起拍价）。
- **失败兜底**：五类上游错误分级映射 + AI 链路 fail-open，确保模型不可用时商家仍可手动发布；调用全程打点可观测。

### 14.3 评测方案与可观测验证
- 业务指标覆盖出价、延时触发、并发峰值、溢价率、GMV、支付成功率，GORM 插件自动采集 DB 指标。
- 增长漏斗链路可观测：提醒点击 → 进房回流 → 点击出价 → 一口价点击 → 购买成功。
- 结构化 JSON 日志经 `X-Request-ID` 跨服务串联、敏感字段脱敏后进 Loki，评委可在 Grafana 验证全链路。
