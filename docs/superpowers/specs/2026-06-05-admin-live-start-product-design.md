# 管理端开播语义与一期产品自洽设计

## 背景

当前管理端商家身份在直播间页面可以看到 `开启直播` 或类似开播按钮。后端已有 `POST /api/v1/live-streams/:id/start`，并已按商家角色和直播间归属做权限校验。该接口会把直播间推进到 `live` 业务状态，从而驱动 H5 直播间、竞拍、一口价、提醒和指标链路。

问题在于：PC 管理端并不具备真实主播推流能力。如果按钮语义表达为“PC 直接开播”，会让产品模型失真。真实市场场景通常是 PC 做经营配置，移动端做主播推流；但本期目标是产品自洽演示版，不引入移动端主播页、推流凭证和流媒体回调。

## 现状（main 代码核对）

本节记录改造前 main 上的真实实现，作为「现状 → 目标」差异基线。

### 开播入口现状

- 开播按钮位于工作台 `Dashboard`，不在直播间详情页 `LiveDetail`。见 [Dashboard.tsx:143-154](file:///Users/bytedance/myself/coding/dy-ai-live-auction-fullstack-cc/frontend/admin/src/pages-new/Dashboard.tsx#L143-L154)。
- 按钮文案当前为 `开启直播`（非本设计目标文案 `开始直播`）。
- 点击后通过 `window.prompt("请输入要开启的直播间 ID")` 让用户手输直播间 ID，再调用 `liveStreamApi.start(id)`，见 [Dashboard.tsx:59-75](file:///Users/bytedance/myself/coding/dy-ai-live-auction-fullstack-cc/frontend/admin/src/pages-new/Dashboard.tsx#L59-L75)。
- 即当前并不存在“在自己的直播间详情页一键开播”“已直播置灰”等交互，本设计的页面交互属于改造目标，不是现状。
- 按钮可见性已按角色隔离：`!isPlatformAdmin` 包裹，管理员不可见，已有回归测试 [Dashboard.roleVisibility.test.tsx](file:///Users/bytedance/myself/coding/dy-ai-live-auction-fullstack-cc/frontend/admin/src/pages-new/__tests__/Dashboard.roleVisibility.test.tsx)。

### 后端鉴权现状（与本设计一致）

- Gateway `POST /api/v1/live-streams/:id/start` 为 `RequireMerchantOnly()`，见 [router.go:118](file:///Users/bytedance/myself/coding/dy-ai-live-auction-fullstack-cc/backend/gateway/router/router.go#L118)。
- auction-service 内部 handler 校验认证、`role==1`（商家）、owner 归属，见 [live_stream_stats.go:32-67](file:///Users/bytedance/myself/coding/dy-ai-live-auction-fullstack-cc/backend/auction/handler/live_stream_stats.go#L32-L67)。

### 关闭/封禁直播现状

- `PUT /api/v1/admin/live-streams/:id/end` 与 `:id/ban` 均为 `RequireAdmin()`，见 [router.go:166-167](file:///Users/bytedance/myself/coding/dy-ai-live-auction-fullstack-cc/backend/gateway/router/router.go#L166-L167)。
- 即现状下商家无法关闭自己的直播间，关闭/封禁都是管理员治理动作。

## 决策

一期采用“PC 管理端触发直播业务状态”的产品口径，按钮文案由现状的 `开启直播` 调整为 `开始直播`，并新增页面说明必须明确：

> 当前版本支持通过 PC 管理端发起直播状态，用于商品讲解、竞拍和一口价交易链路演示；移动端主播推流能力将在后续版本接入。

也就是说，一期的 `开始直播` 不是“PC 端真实推流”，而是“由商家在 PC 管理端发起直播中业务状态，跑通交易闭环”。

## 目标

1. 让管理端开播入口在一期产品语义上自洽。
2. 保持“一商家一个直播间”的业务模型。
3. 商家负责自身直播间经营，管理员只负责治理，不代运营。
4. PC 管理端负责直播配置、商品、竞拍、一口价和状态触发。
5. 不在一期引入移动端主播页、推流凭证、RTMP/HLS 推流链路或流媒体回调。
6. 为二期移动端真实开播保留清晰迁移点。

## 非目标

1. 不实现真实移动端主播页。
2. 不实现推流地址、推流密钥、推流鉴权或推流回调。
3. 不引入直播场次 `LiveSession` 新表。
4. 不重构现有直播间、竞拍、一口价数据模型。
5. 不改变管理员与商家的既有角色枚举。
6. 不允许前端绕过 `gateway-service` 的 `/api/v1` 入口直连后端服务。

## 产品模型

### 一商家一直播间

直播间是商家的长期经营空间。当前 `live_streams.creator_id` 已体现一商家一直播间语义，商家只能操作自己的直播间。

本期默认数据库中的商家账号均已审核通过，不额外实现商家资质审核流程。

### PC 管理端职责

PC 管理端是直播经营控制台，负责：

- 编辑直播间资料。
- 管理商品。
- 创建和管理竞拍。
- 管理一口价商品上下架。
- 发起直播间进入 `直播中` 业务状态。
- 查看自身订单和经营数据。

PC 管理端不负责真实摄像头采集、推流、主播连麦或流媒体链路。

### 移动端主播能力

移动端主播页是二期能力。一期不实现，但产品说明中必须明确后续方向：

- 商家在 PC 管理端完成经营配置。
- 二期主播通过移动端进入直播间。
- 二期开播状态由移动端推流成功或流媒体回调触发。

## 角色权限

| 能力 | 商家 | 管理员 | 说明 |
|---|---:|---:|---|
| 查看直播间列表 | 仅自己的直播间 | 全平台直播间 | 管理员治理视角 |
| 编辑直播间资料 | 仅自己的直播间 | 不允许 | 管理员不代运营 |
| 创建直播间 | 允许 | 不允许 | 商家经营动作 |
| 开始直播 | 仅自己的直播间 | 不允许 | 一期为业务状态触发 |
| 关闭直播 | 不允许（现状） | 可治理关闭 | 现状 `:id/end` 为 `RequireAdmin`，见下方说明 |
| 封禁直播间 | 不允许 | 允许 | 平台治理能力 |
| 一口价上下架 | 仅自己的直播间 | 不允许 | 商家经营动作 |

权限必须由后端兜底，前端只负责改善体验。商家身份来自 JWT 派生的 `X-User-ID`，不能硬编码用户身份或由前端传入 owner ID。

> 关闭直播说明：现状 `PUT /api/v1/admin/live-streams/:id/end` 为 `RequireAdmin()`（[router.go:166](file:///Users/bytedance/myself/coding/dy-ai-live-auction-fullstack-cc/backend/gateway/router/router.go#L166)），商家无法关闭自己的直播间。一期沿用现状、不新增商家关闭直播能力（与「非目标：不重构现有接口」一致）。「商家可结束自己本场直播」属二期能力，需新增商家可调用的结束路由与 owner 校验，见「二期预留」。

## 页面交互设计

### 商家视角

> 改造基线：现状开播入口在工作台 `Dashboard`，通过 `window.prompt` 手输直播间 ID 触发（见「现状」一节）。一期需把开播入口迁移到直播间详情页 `LiveDetail`，针对“当前直播间”一键开播，移除手输 ID 的交互。此为本期前端改造范围。

直播间详情页展示 `开始直播` 按钮。按钮附近展示弱提示：

```text
当前版本支持通过 PC 管理端发起直播状态，用于商品讲解、竞拍和一口价交易链路演示；移动端主播推流能力将在后续版本接入。
```

点击 `开始直播` 后：

1. 若直播间已是 `直播中`，按钮置为弱化或显示 `直播中`。
2. 若直播间未开播，弹出确认文案：

```text
确认开始直播？
当前版本将由 PC 管理端发起直播状态，用于演示观看、竞拍和一口价交易链路；真实移动端推流将在后续版本接入。
```

3. 用户确认后调用 `POST /api/v1/live-streams/:id/start`。
4. 成功后刷新直播间状态，页面展示 `直播中`。
5. 失败时按后端状态码展示错误，不吞错。

推荐按钮文案：

- 未开播：`开始直播`
- 请求中：`开始中...`
- 已直播：`直播中`

### 管理员视角

管理员不展示 `开始直播` 按钮。管理员进入直播间详情页时，页面保持治理语义：

- 可查看直播间资料和状态。
- 可执行封禁直播间。
- 可执行治理性关闭直播。
- 不展示编辑、开播、一口价上下架等商家经营动作。

### 列表页入口

如果直播间列表页存在快速操作，商家可看到 `开始直播`；管理员不可见。列表页和详情页必须共用相同权限判断，避免一个页面隐藏、另一个页面可操作。

## 状态语义

一期继续复用现有直播间状态：

| 状态 | 含义 |
|---|---|
| `0` / 未开播 | 直播间未进入直播中业务状态 |
| `1` / 直播中 | 已由商家发起直播业务状态，可驱动 H5、竞拍、一口价和提醒链路 |
| `2` / 已结束 | 本次直播状态结束 |
| `3` / 已封禁 | 管理员治理封禁 |

需要注意：一期的 `直播中` 表示业务链路进入直播态，不表示服务端已验证真实推流流存在。

> 枚举来源核对：上表与后端 [live_stream.go:9-12](file:///Users/bytedance/myself/coding/dy-ai-live-auction-fullstack-cc/backend/product/model/live_stream.go#L9-L12)（`NotStarted=0 / Live=1 / Ended=2 / Banned=3`）一致。但前端类型注释 [types.ts:118](file:///Users/bytedance/myself/coding/dy-ai-live-auction-fullstack-cc/frontend/admin/src/shared/api/types.ts#L118) 只标注了 `0=未开播, 1=直播中, 2=已结束`，遗漏 `3=已封禁`。一期应顺手补全该注释，避免前端误判封禁态。

二期引入移动端真实推流时，应把 `开始直播` 的最终触发源从 PC 按钮迁移为移动端推流成功或流媒体回调。PC 按钮可退化为 `开播准备` 或 `生成主播开播码`。

## 后端设计

一期沿用现有接口：

- `POST /api/v1/live-streams/:id/start`
- Gateway 路由必须保持 `RequireMerchantOnly`
- Auction service 内部接口必须校验：
  - 已认证用户。
  - 用户角色为商家。
  - 当前用户是直播间 owner。

接口行为：

- 成功：返回 `200` 和 `{ success: true }`。
- 非商家：返回 `403`。
- 非 owner：返回 `403`。
- 无效直播间 ID：返回 `400`。
- 下游业务错误：按现有错误语义返回，不应被 Gateway 统一吞成 `502`。

本期不新增推流相关 API。

## 前端设计

前端应从角色和直播间状态派生按钮可见性：

- `user.role === merchant` 且直播间属于当前商家时，展示 `开始直播`。
- `user.role === admin` 时，不展示 `开始直播`。
- `liveStream.status === live` 时，不允许重复提交，显示已直播状态。
- 请求期间禁用按钮，避免重复点击。

前端文案必须避免暗示 PC 具备真实推流能力。推荐在直播间详情页操作区增加说明卡片或提示文本，不使用弹窗作为唯一解释来源。

## 测试策略

### 前端测试

1. 商家访问自己的未开播直播间，能看到 `开始直播`。
2. 商家点击 `开始直播` 前出现确认文案。
3. 商家确认后调用 `liveStreamApi.start(id)`。
4. 请求中按钮禁用并显示 `开始中...`。
5. 成功后页面状态更新为 `直播中`。
6. 管理员访问直播间详情，不展示 `开始直播`。
7. 已直播状态下不允许重复点击开播按钮。
8. 页面展示一期说明文案。

### 后端测试

现有后端测试已覆盖商家 owner 可开播、非 owner 拒绝、管理员拒绝。若实现中改动接口或路由，需要保持以下回归：

1. `POST /api/v1/live-streams/:id/start` 只允许商家角色。
2. 商家只能启动自己的直播间。
3. 管理员不能调用开播接口。
4. Gateway 透传下游 `4xx`，不把权限错误包装成 `502`。

## 验收标准

1. 商家管理端按钮文案由 `开启直播` 改为 `开始直播`。
2. 开播入口从工作台 `Dashboard` 的 `window.prompt` 手输 ID，迁移为直播间详情页 `LiveDetail` 针对当前直播间一键开播。
3. 页面说明明确当前版本由 PC 发起直播状态，移动端主播推流为后续版本。
4. 商家只能对自己的直播间开始直播。
5. 管理员不展示开播入口，不具备代运营能力。
6. 点击开始直播后 H5、竞拍、一口价、提醒和指标链路仍能进入直播态。
7. 不新增移动端推流、推流凭证或流媒体回调实现。
8. 前端 `types.ts` 直播间状态注释补全 `3=已封禁`。
9. 前端测试覆盖按钮可见性、确认交互和状态更新。

## 二期预留

二期可以在不推翻一期产品语义的前提下演进：

1. PC 管理端从 `开始直播` 演进为 `开播准备`、`生成主播开播码` 或 `邀请主播开播`。
2. 移动端主播页扫码进入商家直播间。
3. 后端签发短期推流凭证。
4. 流媒体服务回调 `stream_connected` 后，服务端再切换到 `直播中`。
5. PC 管理端只展示直播状态和治理动作，不再作为真实开播触发源。
6. 新增商家可调用的“结束自己本场直播”路由（owner 校验），与管理员治理性 `:id/end` 区分；现状 `:id/end` 仍为管理员治理。

## 风险与约束

1. 如果只改按钮不加说明，用户会误解 PC 已具备真实直播能力。
2. 如果管理员仍看到 `开始直播`，会破坏“管理员只治理、不代运营”的角色模型。
3. 如果前端只隐藏按钮而后端不校验 owner，会产生越权开播风险。
4. 如果一期直接引入推流概念但不实现，会导致产品承诺和系统能力不一致。
