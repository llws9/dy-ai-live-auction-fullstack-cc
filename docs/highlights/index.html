02 / 核心亮点 · 旗舰

出价并发：三层控制，而非单点加锁

高并发出价不是「上一把 Redis 锁」这么简单，而是把互斥、版本冲突、事务一致性拆成三层，各解决一类问题。

01
LAYER · MUTEX

Redis 分布式锁

互斥 + 短时限流

SETNX 获取锁，锁值用纳秒时间戳防误删；解锁/续期走 Lua 脚本保证原子。TTL 5s，默认重试 10 次 × 50ms。

02
LAYER · CAS

DB 乐观锁

防覆盖

价格更新走 UPDATE ... WHERE id=? AND version=? 的 CAS 路径，RowsAffected=0 视为冲突，自动「重读→重校验→重试」最多 3 次。

03
LAYER · TXN

单事务一致性

结果一致

bid 写入与竞拍状态更新位于同一事务；进入临界区前后各做一次最低价校验，避免排队过程中的脏写。

核心取舍：延时判定、通知发送、点天灯自动出价等高成本逻辑全部移到锁外异步执行 —— 把锁的职责收敛到「保护共享状态」，而非「串行化整个业务流程」。
03 / 核心亮点

WebSocket：完整的实时同步链路

重点不在「用了 WebSocket」，而在于把连接、恢复、广播和背压处理做成一条完整链路，在高频更新、断线重连和慢连接并存时仍然可控。

RM

房间模型清晰

竞拍房 / 弹幕房 / 用户通知房三类独立管理，广播粒度按业务域拆开；慢客户端采用非阻塞丢弃，避免单连接拖慢整个房间。

RC

状态恢复可落地

服务端心跳 30s Ping / 60s Pong 超时；前端指数退避重连 [1,2,4,8,16,30]s，4401 鉴权失败单独处理；重连后 sync_request 回灌当前价、领先者、结束时间。

TH

高频广播可控

前端按类型节流（rank_update 200ms / bid_placed 100ms），服务端排名广播同样窗口节流，并通过房间归属校验避免消息串台。

目标不是单纯「把数据推到前端」，而是让实时链路具备「可连接、可恢复、可控广播」的完整工程闭环，而非只能在理想网络下工作的演示级实现。
04 / 核心亮点

测试能力：独立的编排系统，而非附属脚本

项目没有把测试停留在单测和本地脚本层面，而是单独实现了一套可在线触发、可观测、可复盘的测试编排平台。

05 / 核心亮点

一口价：单服务闭环 + 三层并发控制

一口价不是在竞拍主路径上加一个「直接成交」分支，而是在 auction-service 内单独实现一套 fixed_price 闭环。

①

Redis Lua 原子预扣

库存键与「已购用户集合」在同一段 Lua 脚本内完成初始化检查、重复购买判断、库存扣减、已购标记，从机制上避免超卖与重复抢购。

②

DB 事务保一致

余额扣减走 available_amount >= amount 条件更新，购买记录写入与资金扣减位于同一事务，避免读-判-写竞态。

③

唯一索引 + 幂等 + 补偿

(item_id, user_id) 唯一索引兜底重复提交；前端携带 X-Idempotency-Key；Redis 预扣成功但 DB 失败则补偿回滚。

设计价值：把固定价抢购与竞拍出价彻底解耦。竞拍主链路不必为一口价承担额外状态复杂度，而直播间又能同时承载「竞拍 + 固定价抢购」两类交易模型。
06 / 核心亮点

用户触达：实时推送 + 未读落库 + 去重回执

通知能力不是简单「发一条消息」，而是按时效性和可追溯性拆成多条链路，解决实时性、离线可追溯与重复提醒三个常见问题。

RT

被超价实时推送

出价成功后若前任领先者被超越，立即写入通知并通过用户专属 WS 房间推送 bid_outbid。

DB

结果通知落库沉淀

竞拍结束时，中标与未中标结果统一进入通知中心，确保用户离线时也不会错过关键结果。

LV

提醒场景分层

关注主播的开播提醒走 pending-reminder 与冷启动调度器；订阅商品的开拍提醒走 reminder 表落库再热拉补齐。

ID

幂等由回执表保证

开播提醒用 (user_id, live_stream_id, live_started_at) 去重，单品提醒用 (user_id, auction_id) 去重，避免重复触达。

07 / 核心亮点

消息链路：热拉冷推分层，而非一律依赖 WebSocket

项目没有把所有消息都强行塞进 WebSocket，而是按时效性和触达成本做分层 —— 把「实时性」和「广播成本」分开考虑。

热数据 · 推送
WebSocket 实时广播
竞拍排名、价格变化、实时通知等强时效数据，前后端两侧均做节流控制。
⇄
冷数据 · 拉取
hot-pull 主动补齐
用户回到前台时主动调用 hot-pull 与未读数接口，补齐离线期间的提醒，而非「在线才收到」。
⇄
冷门 · 定向
ColdPushScheduler
按 Redis 热/冷集合做定向提醒，避免把所有直播都当广播事件处理。
恢复关注的是状态，不是日志重放：断线恢复时从 Redis 读取竞拍快照，仅同步业务所需的最小状态，而非回放完整消息流。

日 / 夜外观：全局主题系统，而非局部样式切换

H5 端深浅色切换不是零散 CSS 改造，而是围绕主题系统做的基础设施建设。本页右上角的切换按钮即是同款能力的活体演示。

09 / 关键工程难点

四大难点与解决方案

竞拍系统的复杂度集中在状态正确性、防狙击吞吐、外部依赖降级与自动代理出价的边界控制。

5 态有向图，状态转换集中在纯判断模块中，避免条件散落到 handler/service/定时任务。事件驱动（出价触发延时/封顶）与时间驱动（scheduler 每秒扫描开收场）共享同一套状态约束。金额统一用 shopspring/decimal，避免 float 比较误差。

难点二 · 防狙击自动延时

出价落在结束前窗口（ShouldTriggerDelay）且累计延时未超上限（CanDelay）时切「延时中」。时间更新用 SQL DATE_ADD(end_time, INTERVAL ? SECOND) 原子累加；延时判定放到锁外，避免放大锁竞争。

难点三 · 外部依赖降级兜底

Redis 锁异常退化到本地 sync.Map 锁（LoadOrStore 原子 CAS）；弹幕限流故障走放行；广播队列满非阻塞丢弃；AI 链路默认 fail-open。原则：不把「依赖可用」当作「业务可用」前提。

难点四 · 点天灯（自动代理出价）的锁嵌套与递归触发

业务语义接近 eBay 代理出价：用户提交预算上限后，被超越时按最小加价自动跟价。实现要点：

补偿替代长事务
短事务创建订阅 → 事务外执行出价 → 失败则补偿删除订阅，避免出价逻辑与 Redis 锁/事务/WS 推送包进一个长事务。
+
并发粒度细化
锁粒度从 per-auction 拆到 per-subscription，配合容量信号量限制并发，防止热点竞拍下自动出价互相阻塞。
+
递归显式阻断
通过 SkipSkyLampTrigger 等标记避免自动出价再次触发自身，防止陷入自激循环。
10 / AI 能力使用

把模型调用约束成可替换、可降级的工程能力

重点不是「接了一个模型」，而是把多模态生成约束成结构化、可替换、可降级的工程能力。

MD

模型选型

火山方舟豆包 doubao-seed-1-6-vision-250815，OpenAI 兼容协议接入，能力形态为视觉多模态生成。

IO

接入位置

商家后台「商品 AI 一键文案」。输入最多 6 张商品图 + 关键词，输出标题、描述、卖点和建议起拍价草稿。

PR

Prompt 设计

System prompt 约束扮演「直播竞拍文案专家」并显式定义输出 schema，开启 JSON mode 保证结构化稳定，temperature 0.6，图文混合多模态输入。

FB

失败分级双兜底

服务端对超时/上游错误/解析失败分级映射为 400/429/502/504；前端对图片数量、格式、错误码文案再做一次用户态兜底。

11 / 可观测性

从请求、业务指标到实验效果的统一观测面

可观测性不是「出了问题再看日志」，而是从请求、业务指标到实验效果都有统一观测入口。

LOG

结构化日志

统一 JSON LogEntry，覆盖 service/request_id/operation_type/user_id/duration；X-Request-ID 跨服务串联，password/token/secret 统一脱敏。进 Loki 保留 31 天，Grafana LogQL 查询。

MET

业务指标完备

Prometheus 覆盖竞拍创建、出价、延时触发、并发峰值、溢价率、GMV、支付成功率、WS 连接数、SQL 耗时；GORM 插件自动采集 DB 指标，前端埋点回流同一大盘。

AB

实验能力闭环

GrowthBook Go SDK，网关侧特征预评估，X-Experiment-Context 透传下游；曝光与转化统一落 Prometheus，形成从分流到结果观测的闭环。

问题
原有监控只覆盖接口、出价、一口价等交易结果，缺少「提醒点击、进房、点击出价、购买成功」等漏斗中间节点。只看成交，无法判断用户损失发生在触达、回流还是交易阶段。

14 / 设计亮点

H5 首页维度重构：从「竞拍维度」到「直播间维度」

一次以第一性原理驱动的信息架构重构：不打补丁，而是追问「首页到底该回答用户什么问题」，从根上修正列表维度错配。

问题 · 维度错配
首页原按竞拍维度铺卡，但 live_streams.CreatorID 唯一 → 一商家一直播间，且会累积大量已结束竞拍。导致历史「已结束」一条条铺成独立卡，把活跃竞拍挤到下方。
洞察 · 伪问题
「已结束如何分页」是伪问题。根因是把已结束当成了一等卡片。改为直播间维度、明确已结束仅作氛围信号后，复杂分页不复存在。
价值 · 修根因
没有为「已结束分页」打补丁，而是修正列表维度本身，复杂查询随之消失；与「一商家一直播间」业务模型对齐。

主动界定已完成与未完成的边界

以下属于「能力已部分实现但主流程未接通」或「当前确实不存在」的范围，答辩与文档中按现状描述，避免把局部实现表述成完整闭环。

CAUTION
保留本节的目的不是自我降级，而是用边界意识换取可信度 —— 评审更信任「敢于说清楚没做什么」的项目。
起拍价 StartPrice：当前仅作展示字段与规则元数据存储，出价校验路径未读取；实际行为更接近「0 元起拍 + 首次最低出价 = 加价幅度」。
服务端时间同步：time_sync 广播与 useServerTime 校准已实现，但主直播页 LiveRoomSlide 仍基于本地时钟 + 服务端 end_time 计算倒计时，尚未接入 offset 校准结果。
出价提示音：代码未实现音效反馈，当前感知来自 toast、震动、动画与视觉高亮，不应表述为「已有提示音」。
Admin 静态页：竞拍规则模板页 AuctionRules 与权限页 Permissions 仍是静态 Mock，未接后端。需区分「后端 RBAC 已实现」与「页面已联通」两个层次。
product 服务指标：当前暴露 /metrics 的是 gateway 与 auction，product 服务尚未接入 Prometheus 指标导出。
