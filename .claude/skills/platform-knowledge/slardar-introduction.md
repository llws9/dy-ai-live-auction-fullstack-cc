## Slardar 是什么？

**Slardar** 是 ByteDance 内部广泛使用的一款**应用性能监控 (APM) 平台**。

对于前端开发而言，Slardar 的核心价值在于**“看见”线上应用的真实表现**。当用户的浏览器加载并运行我们的代码时，可能会遇到各种各样的问题，例如：

-   代码在特定浏览器或网络环境下报错。
-   图片、CSS 或 JS 文件加载失败。
-   API 请求异常或响应缓慢。
-   页面长时间白屏，用户无法操作。
-   交互操作卡顿，响应迟钝。

这些问题发生在用户的设备上，我们通常难以复现和定位。Slardar 通过在前端应用中植入一个轻量级的 SDK (`@slardar/web`)，实时采集这些异常、性能和行为数据，并将其上报到数据中心。开发者可以在 Slardar 的可视化后台，直观地查看报表、分析趋势、定位问题，从而持续提升应用的质量和用户体验。

## 前端监控能做什么？

通过接入 Slardar Web SDK，你可以开箱即用地获得以下监控能力：

| 数据类型 | 描述 | 解决的问题 |
| :--- | :--- | :--- |
| **JS 错误** | 自动捕获全局未处理的 JavaScript 异常。 | 快速发现导致页面功能异常或崩溃的代码缺陷。 |
| **资源错误** | 监控图片、脚本、样式表等静态资源的加载失败。 | 定位因 CDN 问题、路径错误或网络策略导致的资源加载问题。 |
| **Promise 异常** | 捕获未被 `catch` 的 Promise `reject`。 | 发现异步流程中的潜在逻辑错误。 |
| **网络请求** | 自动监控 `fetch` 和 `XMLHttpRequest` 的成功、失败和耗时。 | 分析 API 健康度，定位慢请求或失败请求。 |
| **页面性能** | 采集核心 Web 指标 (LCP, CLS, INP) 等性能数据。 | 量化并优化用户感知的加载速度、交互响应和视觉稳定性。 |
| **白屏与冻结** | 监测页面是否出现长时间无内容或无响应的情况。 | 发现并解决最严重的渲染和执行阻塞问题。 |
| **用户行为** | 记录页面跳转、点击等关键操作，形成“面包屑”(Breadcrumb)。 | 在错误发生时，提供复现问题的上下文路径。 |
| **自定义日志** | 允许你通过 API 主动上报业务相关的事件或异常。 | 监控关键业务流程的成功率，或捕获特定逻辑分支的错误。 |

## 接入前需要准备什么？

在开始技术接入之前，你需要完成以下准备工作：

1.  **申请 `bid`**
    -   `bid` (Business ID) 是你的应用在 Slardar 平台上的唯一标识。所有的监控数据都会归属到这个 `bid` 之下。
    -   你需要**在 Slardar 控制台为你的项目创建一个应用**，创建成功后即可获得对应的 `bid`。

2.  **明确上报域名**
    -   Slardar SDK 需要将数据上报到一个指定的接收端域名。这个域名通常是**由平台统一分配的**，并且在不同区域（如国内、海外）会有所不同。
    -   在初始化 SDK 时，你需要根据你的应用部署的地理位置选择正确的区域配置，SDK 会自动使用该区域对应的上报域名。

3.  **团队角色分工**
    -   通常，项目的 **Tech Lead** 或**主要负责人**负责申请 `bid` 和在 Slardar 平台上进行基础配置（如告警规则、采样率等）。
    -   **前端开发同学**则负责在代码中集成 SDK，并根据业务需求进行自定义上报。

准备好 `bid` 后，你就可以进入下一章节，开始激动人心的接入之旅了！
