## 1. Key 命名规范与分层

良好、一致的 Key 命名是国际化项目可维护性的基石。混乱的 Key 命名会给后续的文案查找、修改和管理带来巨大困难。

**最佳实践：**

- 采用分层结构：使用点 `.` 作为分隔符，建立清晰的层级关系。推荐的结构是 `[页面/模块].[组件/区域].[具体元素]_[状态/类型]`。
  - 示例 1（登录页）：`login.form.username.placeholder` -> “请输入用户名”。
  - 示例 2（公共按钮）：`common.button.submit` -> “提交”。
  - 示例 3（带状态的提示）：`profile.update.toast_success` -> “用户信息更新成功！”。

- 语义清晰，避免歧义：Key 的名称应能清晰地反映其在 UI 中的位置和功能。避免使用过于泛化或模棱两可的名称，如 `text1`、`message`、`temp_key`。

- 避免将文案内容作为 Key：不要使用源文案（尤其是长句）直接作为 Key。这会导致 Key 过长、难以管理，并且一旦源文案变更，Key 也需要随之变更。
  - 错误示例：
    ```ts
    t('欢迎来到我们的网站')
    ```
  - 正确示例：
    ```ts
    t('homepage.hero.title')
    ```

- 统一风格：在整个团队内约定并遵守统一的命名风格，例如全部使用小写字母、下划线 `_` 或驼峰式 `camelCase`。

## 2. 占位符与插值

当文案中包含动态数据（如用户名、数字）时，必须使用插值，而不是在代码中拼接字符串。

**最佳实践：**

- 使用命名占位符：在 Starling 平台和代码中，使用有意义的占位符名称，遵循 ICU 语法。
  - 平台文案：`你好, {userName}，你有 {count} 条未读消息。`。
  - 代码调用：
    ```ts
    t('dashboard.welcome', { userName: '小明', count: 5 })
    ```

- 严禁拼接字符串：绝对不要在代码中通过 `+` 或模板字符串来拼接翻译片段。这会破坏句子在不同语言中的语法结构，导致翻译错误或不自然。
  - 错误示例：
    ```ts
    t('common.hello') + userName + t('common.welcome')
    ```
  - 正确示例：
    ```ts
    t('greeting.full', { name: userName })
    // 平台文案：你好 {name}，欢迎回来。
    ```

- 考虑语法差异：不同语言的语序和语法规则差异很大。一个完整的句子单元能让翻译人员根据目标语言的习惯自由调整语序。
  - 例：英文中动词在前，德语中动词可能在后。如果将句子拆分为多个 Key，翻译将变得极其困难。

## 3. 文案设计注意事项

前端开发者在实现 UI 时，应与产品和设计师沟通，确保文案设计本身对国际化友好。

- 考虑文本长度：不同语言的同一句话，长度可能差异巨大（例如，德语通常比英语长 30%）。UI 设计需要预留足够的空间，避免文本溢出或换行导致布局错乱。

- 复用性与上下文：设计可复用的文案片段时要格外小心。一个词在不同上下文中可能有完全不同的翻译。
  - 例：“地址”可以指“家庭地址”，也可以指“URL 地址”。应创建两个独立的 Key，如 `profile.form.home_address` 和 `settings.api.endpoint_address`，并为翻译人员提供上下文说明。

- 性别与复数变体：如果文案涉及性别或数量，务必使用 ICU 的 `plural` 和 `select` 语法，而不是在代码中用 `if/else` 判断。这能保证在不同语言（如阿拉伯语有多种复数形式）中正确显示。

- 避免在图片中嵌入文字：图片中的文字无法被 Starling 工具扫描和翻译，应尽量将文字作为独立的 HTML 元素叠加在图片之上。

## 4. 性能与用户体验优化

- 利用缓存：`@ies/starling_client` 默认开启 `localStorage` 缓存（以官方文档为准）。确保不要无意中禁用它。对于后续访问，这将极大地提升加载速度。

- 批量与按需加载：对于大型应用，一次性拉取所有文案可能会影响性能。可以考虑按需加载策略：
  - 在应用启动时只加载公共和首屏所需的文案。
  - 在路由切换时，异步加载新页面所需的文案模块（需要 i18next 的 `ns` 功能支持）。

- 首屏优化（SSR/SSG）：对于需要做 SEO 或对首屏性能要求高的应用，务必采用服务端渲染或静态站点生成。
  - 在服务端预取文案并将其注入 HTML，避免客户端二次请求。
  - 将首屏必需的文案打包进初始 bundle，非必需的文案进行懒加载。

- 懒加载（Lazy Loading）：对于不立即显示的组件（如模态框、下拉菜单），可以延迟加载其对应的文案资源，减少主包体积。

## 5. SSR/CSR 差异与处理

- 首屏一致性：这是 SSR 应用中最核心的问题。必须确保服务端渲染出的 HTML 和客户端“激活”（Hydration）后的内容完全一致。否则，React/Vue 会发出警告，甚至可能导致 UI 错乱。

- Hydration 策略：
  1. 服务端：通过 `@ies/starling_node` 获取当前请求语言的文案（以官方文档为准）。
  2. 数据注入：将获取到的文案和当前语言 `locale` 作为 `props` 注入到页面组件中。
  3. 客户端：i18n 库初始化时，直接使用从 `props` 传入的文案数据，跳过客户端的首次网络请求。

- 回退策略与监控：
  - 在 SSR 场景下，如果服务端从 Starling 拉取文案失败，应有一套明确的回退策略（例如，使用构建时打包的旧版本文案，或直接渲染兜底语言），同时记录错误日志。
  - 在客户端，监控 `starlingClient.load()` 的 `catch` 块，捕获文案加载失败的异常，并上报给监控系统，以便及时发现线上问题（以官方文档为准）。

## 6. 推荐工作流

一个高效、可靠的前端国际化工作流应包含以下环节：

1. 开发阶段：正常进行功能开发，可使用中文硬编码。完成功能后，运行 `@ies/starling-cli` 的 `scan` 和 `upload` 命令，将新文案提交至 Starling 平台（以官方文档为准）。

2. 翻译阶段：通知翻译或运营团队在平台进行翻译和审核。

3. 联调阶段：
   - 运行 `starling replace` 将代码替换为 `t(key)` 的形式（以官方文档为准）。
   - 在测试或灰度环境下拉取已翻译的文案进行功能和 UI 的联调。
   - 使用伪本地化：通过特定配置（如在 URL 添加 `?lng=pseudo`），让所有文案显示为带特殊字符的扩展形式（如 `[¡Ĥéļļó Ŵóŕļð!]`），可以快速暴露以下问题：
     - 未被国际化处理的“漏网之鱼”文案。
     - UI 布局是否能容纳更长的字符串。

4. 测试阶段：QA 团队在多语言环境下进行完整的回归测试，特别关注布局、格式（日期、数字）和复数等场景的正确性。

5. 发布阶段：在 Starling 平台将文案“发布”到生产环境，CI/CD 流水线构建和部署前端应用。

## 7. 常见“陷阱”与排错清单

- 动态 Key：
  - 陷阱：使用变量拼接 Key，如 `t(`page.title_${tab}`)`。
  - 后果：`@ies/starling-cli` 无法静态扫描到这些 Key，导致文案丢失（以官方文档为准）。
  - 解决：使用完整的 Key，将动态部分作为 `select` 或其他逻辑判断。如果必须动态，请确保这些 Key 能在平台上手动维护。

- 标点与空格：
  - 陷阱：在代码中为翻译文本手动添加标点或空格，如 `t('common.greeting') + '!'`。
  - 后果：破坏了某些语言的语法规则（如法语中感叹号前需要一个空格）。
  - 解决：将标点和空格作为翻译内容的一部分，交由翻译人员处理。

- HTML/富文本：
  - 陷阱：文案中包含 HTML 标签，直接使用 `t()` 函数渲染。
  - 后果：HTML 标签被当做普通字符串转义，无法正确渲染。
  - 解决：使用 `react-i18next` 的 `<Trans>` 组件或 Vue 的 `v-html` 指令，并确保内容安全。

- RTL/LTR（从右到左/从左到右）：
  - 陷阱：忽略了阿拉伯语、希伯来语等 RTL 语言的布局需求。
  - 后果：布局在 RTL 语言下完全错乱。
  - 解决：在根元素上根据当前语言动态添加 `dir="rtl"` 或 `dir="ltr"` 属性，并使用逻辑 CSS 属性（如 `margin-inline-start` 代替 `margin-left`）。

- 日期/单位格式：
  - 陷阱：在代码中硬编码日期格式（`YYYY-MM-DD`）或货币单位（`$`）。
  - 后果：在不同文化区域显示不正确。
  - 解决：使用 ICU 格式化或 `Intl` API，让浏览器或 i18n 库根据当前 `locale` 自动处理。
