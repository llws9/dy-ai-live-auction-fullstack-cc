## 合理设置采样与限流

对于高流量的应用，100% 的数据上报是不切实际且没有必要的。你需要制定合理的采样策略。

-   **生产环境 (Production)**
    -   **目标**：在控制成本的同时，确保能捕捉到具有代表性的问题。
    -   **建议**：
        -   对于**错误 (Error) 日志**，可以设置较高的采样率，甚至 `1` (100%)，因为错误通常是低频且高优的。
        -   对于**性能 (Performance) 和自定义事件 (Event) 日志**，可以根据应用流量设置一个较低的采样率，例如 `0.1` (10%) 或 `0.01` (1%)。
        -   Slardar 平台通常支持**分类型采样**，允许你为不同类型的日志（如 `error`, `performance`, `log`）设置独立的采样率。请在 Slardar 控制台进行配置。
        -   如果平台不支持分类型采样，可以在 `init` 时设置一个全局的 `sampleRate` 作为基准。

-   **开发与测试环境 (Development / Staging)**
    -   **目标**：尽可能多地发现问题。
    -   **建议**：采样率设置为 `1` (100%)，以确保所有开发和测试期间产生的数据都被记录下来。同时，在开发环境可以使用 `debug: true` 来实时查看上报内容，便于调试。

## 保护用户隐私与数据脱敏

在监控的同时，保护用户隐私是我们的首要责任。

-   **禁止上报原始敏感信息**
    -   **绝对不要**通过 `setUser` 或 `setContext` 上报任何可以直接识别个人身份的原始信息，例如：姓名、身份证号、手机号、家庭住址、银行卡号等。
    -   应使用内部系统的、经过脱敏的唯一标识符，如 `user_id`。

-   **利用 `beforeSend` 钩子进行数据清洗**
    -   在 `init` 配置中实现 `beforeSend` 函数，作为数据上报前的最后一道防线。
    -   在这个函数里，检查即将上报的 `event` 对象，对其中可能意外混入的敏感信息进行**审查和替换**。

    ```typescript
    browserClient('init', {
      bid: 'YOUR_BID',
      beforeSend: (event) => {
        // 示例：移除 network 请求中的敏感 header
        if (event.request?.headers?.['x-user-token']) {
          event.request.headers['x-user-token'] = '[REDACTED]';
        }

        // 示例：检查自定义上报的额外信息
        if (event.extra?.userInfo?.phone) {
          // 如果发现敏感信息，可以选择脱敏或阻止上报
          // return null; // 直接阻止
          event.extra.userInfo.phone = '[REDACTED]'; // 脱敏处理
        }
        
        return event;
      }
    });
    ```

## 善用标签化进行数据隔离与筛选

标签 (Tag)，在 Slardar 中通常体现为 `dimensions` 或上下文信息，是实现数据精细化分析的关键。

-   **`env` (环境)**
    -   务必在 `init` 时设置 `env` 标签 (`'production'`, `'staging'`, `'development'`)。这是隔离不同环境数据的最基本、最重要的方式。

-   **`release` (版本)**
    -   务必设置 `release` 标签，值为每次发布的唯一标识（如 Git commit hash）。
    -   **价值**：
        -   当新版本上线后，可以筛选出只在新版本中出现的错误。
        -   当修复一个错误后，可以对比新旧版本的数据，验证修复是否生效。
        -   可以快速回溯某个历史版本引入的问题。

-   **`user_id` (用户标识)**
    -   通过 `setUser({ id: '...' })` 设置用户 ID。
    -   **价值**：
        -   当收到用户反馈问题时，可以根据其 `user_id` 查询他遇到的所有错误和性能问题，快速复现场景。
        -   可以统计受某个特定错误影响的用户数量。

-   **自定义业务标签**
    -   通过 `setContext` 或 `reportEvent` 的 `dimensions` 参数，添加与你的业务紧密相关的标签。
    -   **示例**：
        -   `business_area`: 'Order', 'ProductDetail', 'ShoppingCart' (业务模块)
        -   `user_type`: 'FreeUser', 'PaidUser' (用户等级)
        -   `ab_test_group`: 'group_A', 'group_B' (A/B 测试分组)
    -   **价值**：可以让你从业务维度深入分析问题。例如，“订单模块的 JS 错误率是否高于其他模块？”或“B 组实验用户的页面加载性能是否优于 A 组？”

## 性能观测与优化建议

Slardar 自动采集的核心 Web 指标是衡量和优化用户体验的基石。

-   **关注核心指标趋势**
    -   定期在 Slardar 平台上查看 **LCP** (加载性能)、**INP** (交互性能) 和 **CLS** (视觉稳定性) 的 P75/P95 分位值趋势。指标的突然恶化通常意味着新上线的版本存在性能问题。

-   **结合 `release` 版本号分析**
    -   当发现性能指标劣化时，立即关联 `release` 版本号，定位到是哪个版本引入的问题，从而快速回滚或修复。

-   **利用自定义事件进行深度性能分析**
    -   对于一些框架未覆盖的、业务特定的长任务，使用 `reportEvent` 手动上报其耗时。

    ```typescript
    // 场景：一个复杂的前端数据处理任务
    const startTime = Date.now();
    processLargeDataset(data);
    const duration = Date.now() - startTime;

    browserClient('reportEvent', 'ClientDataProcessing', {
      duration: duration,
    }, {
      dataset_size: data.length,
    });
    ```
    -   通过这种方式，你可以监控到那些隐藏在业务逻辑深处的性能瓶颈。

## 团队协作与告警联动

Slardar 不仅仅是一个工具，更应融入团队的研发流程中。

-   **配置告警规则**
    -   在 Slardar 平台上，针对关键指标设置告警阈值。例如：
        -   JS 错误率在 5 分钟内突增 200%。
        -   生产环境出现新的（未见过的）JS 错误。
        -   LCP 的 P75 值超过 3 秒。
    -   将告警通知发送到团队的即时通讯群组（如飞书、Slack）。

-   **建立值班 (On-call) 制度**
    -   明确指定一个或一组同学作为值班工程师，负责响应和初步处理 Slardar 告警。
    -   值班工程师的职责是：
        1.  收到告警后，第一时间判断其严重性和影响范围。
        2.  在 Slardar 平台上查看错误详情、影响版本、用户分布等信息。
        3.  如果是已知问题或误报，进行标记和处理。
        4.  如果是新问题，快速定位到相关的业务或技术负责人，并创建缺陷跟踪单。

-   **定期复盘 (Review)**
    -   每周或每个迭代周期，团队一起回顾 Slardar 报表，分析 Top 错误、性能瓶颈和用户反馈问题。
    -   将解决线上问题作为常规研发任务的一部分，持续迭代，形成“发现问题 -> 分析定位 -> 修复验证”的闭环。
