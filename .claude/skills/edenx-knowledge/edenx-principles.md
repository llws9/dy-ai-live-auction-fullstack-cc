## 3. EdenX 核心原理

EdenX 的强大功能背后，是一套清晰且先进的设计原理。理解这些原理，有助于开发者更深入地掌握 EdenX，并充分利用其优势。 EdenX 的核心可以概括为三大组件：构建器 (Builder)、运行时 (Runtime) 和服务端 (Server)。::cite[3]

### 3.1. 构建器 (Builder)：高性能与高灵活性的基石

构建器是 EdenX 的心脏，负责将源代码编译、打包成可在浏览器运行的产物。::cite[3]

*   **融合与演进:** EdenX 的构建能力源于对内部两大框架 Eden 和 Jupiter 的构建核心的整合，并在此基础上持续演进。::cite[9] 它将构建核心（EdenX Builder）设计为独立组件，可以脱离 EdenX 框架单独使用。::cite[9]

*   **双引擎驱动:** EdenX Builder 在架构上支持多种打包工具。它默认使用 Webpack 5，以保证生态的稳定和兼容性；同时，它也支持一键切换到基于 Rust 的高性能打包工具 Rspack。EdenX Builder 负责抹平两者之间的配置差异，让业务项目可以低成本地迁移到 Rspack，享受极致的构建性能。::cite[9]

*   **插件化扩展:** 构建过程是高度可扩展的。通过插件机制，开发者可以轻松地介入构建流程，例如修改 Webpack/Rspack 配置、添加自定义的加载器 (Loader) 或插件，以满足各种定制化需求。::cite[4]

### 3.2. 运行时 (Runtime)：赋予应用生命力

运行时提供了应用在浏览器端运行所需的一系列核心功能，它以插件化的方式组织，开发者可以按需选用。::cite[3]

*   **核心功能:** 运行时提供了路由管理（基于 React Router）、状态管理、数据请求、生命周期管理、渲染模式支持（CSR/SSR）等。::cite[3]
*   **内部生态集成:** 它无缝集成了公司内部的多种基础库，如 Slardar (监控)、Tea (遥测)、Garfish (微前端) 等，开发者无需手动配置即可使用这些成熟的解决方案。::cite[3]
*   **插件化设计:** 运行时的各项功能（如路由、状态管理）都是通过插件来实现的。开发者也可以编写自己的运行时插件，来修改 React 组件的渲染逻辑或扩展应用能力。::cite[4]

### 3.3. 服务端 (Server)：一体化开发的核心

EdenX 提供了一个统一的 Web Server，贯穿了开发和生产环境，为一体化开发体验提供了保障。::cite[3]

*   **统一服务:** 无论是 `edenx dev` 启动的开发服务器，还是生产环境部署的服务，都基于同一个 Server 核心，确保了环境的一致性。
*   **BFF 层:** 内置了 BFF (Backend for Frontend) 能力。开发者可以在项目中创建 API 目录，并像写普通函数一样编写服务端接口。EdenX 会自动将其转换成可供前端调用的 HTTP 接口，极大地简化了全栈开发。
*   **SSR 支持:** 开箱即用地支持服务器端渲染 (SSR)。EdenX 处理了所有复杂的同构逻辑，开发者只需关注业务代码，即可实现高性能的、对 SEO 友好的 SSR 应用。::cite[3]

### 3.4. 渐进式原理

“渐进式”是 EdenX 的核心设计哲学之一。这意味着开发者可以根据项目的实际情况，自由选择使用 EdenX 的深度。::cite[6]

1.  **作为纯构建工具:** 你可以仅使用 EdenX 的构建能力来打包一个标准的 React 应用，而不使用任何运行时功能。
2.  **按需启用运行时:** 在构建的基础上，可以逐步启用 EdenX 的运行时插件，如约定式路由、状态管理等。
3.  **拥抱一体化:** 进一步地，可以启用 SSR 和 BFF 功能，享受 EdenX 带来的一体化全栈开发体验。

这种设计避免了传统框架“全家桶”式的笨重，让框架能够灵活适应从小型项目到大型复杂应用的各种需求。
