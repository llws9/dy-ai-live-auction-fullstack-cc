# EdenX 渲染模式深度解析

在现代 Web 开发中，如何将应用内容呈现给用户，即“渲染模式”，是决定应用性能、用户体验和 SEO 效果的关键因素。EdenX 框架提供了灵活且强大的渲染能力，支持从传统的客户端渲染（CSR）到先进的服务端渲染（SSR）、静态站点生成（SSG）乃至 React Server Components（RSC）等多种模式。

本文档将对 EdenX 支持的各种渲染模式进行全面介绍，帮助你理解它们的核心特点、适用场景，并为你选择最适合项目的渲染策略提供决策依据。

## 渲染模式速览

下表快速对比了 EdenX 支持的主要渲染模式：

| 渲染模式                  | 核心特点                                                 | 适用场景                                         |
| ------------------------- | -------------------------------------------------------- | ------------------------------------------------ |
| **CSR (客户端渲染)**      | 在浏览器端执行 JavaScript，动态生成和更新 DOM。            | 交互性极强、对 SEO 要求不高的应用（如后台管理系统）。 |
| **SSR (服务端渲染)**      | 在服务器上预先将页面渲染成完整的 HTML 字符串后返回给浏览器。 | 对首屏加载速度和 SEO 有较高要求的网站（如内容型网站、电商）。 |
| **Streaming SSR (流式 SSR)** | SSR 的进阶版，边在服务端渲染边将 HTML 片段流式传输到浏览器。 | 需要更快首屏可交互时间的应用（**这是 EdenX 中 SSR 的默认模式**）。 |
| **SSG (静态站点生成)**    | 在**构建时**就为每个页面生成一个静态的 HTML 文件。        | 内容更新不频繁、追求极致加载速度的网站（如博客、文档站、营销页）。 |
| **RSC (React Server Components)** | 允许部分组件完全在服务端运行，其代码不发送到客户端。     | 追求极致性能、希望大幅减少客户端 JavaScript 体积的项目。 |

## 性能对比与技术演进

从 SSR 到 Streaming SSR，再到结合 RSC，渲染技术的演进主要围绕着两个目标：**更快的首屏内容呈现**和**更少的客户端 JavaScript**。

- **SSR**：解决了 CSR 首屏白屏和 SEO 的问题，但其瓶颈在于服务器必须等待所有数据获取完毕、整个页面完全渲染后才能响应，导致 **TTFB（Time to First Byte，首字节时间）** 可能较长。

- **Streaming SSR**：通过流式传输，服务器可以立即发送页面的骨架（如 `<head>` 和导航栏），然后随着数据和组件的就绪，陆续发送页面的其余部分。这使得用户能更快地看到页面内容，显著改善了**感知性能**，但客户端需要加载的 JavaScript 总量并没有减少。

- **Streaming SSR + RSC**：这是目前最前沿的方案。它结合了流式渲染的快速响应和 RSC 的轻量化优势。由于 Server Components 的代码和逻辑停留在服务端，最终发送到客户端的只是渲染好的 HTML 和少量用于交互的 Client Components 的 JavaScript。这从根本上减少了客户端的负担，实现了更快的加载和交互。

## 如何选择渲染模式

选择合适的渲染模式需要综合考虑项目的业务需求、性能目标和开发复杂度。

### 快速决策指南

-   **内容静态、更新频率低？** -> **SSG (静态站点生成)**
    -   *示例*：个人博客、产品文档、公司官网。
    -   *优势*：无需服务器动态计算，可以直接部署在 CDN 上，访问速度最快，成本最低。

-   **需要 SEO，且首屏性能至关重要？** -> **Streaming SSR (流式服务端渲染)**
    -   *示例*：新闻门户、电商列表页、社区论坛。
    -   *优势*：EdenX 的默认 SSR 模式，开箱即用，提供了优秀的 SEO 和首屏性能平衡。

-   **追求极致性能，并希望挑战前沿技术？** -> **Streaming SSR + RSC**
    -   *示例*：大型、复杂的、对性能有苛刻要求的 C 端应用。
    -   *优势*：最大程度减少客户端 JavaScript，实现最快的加载和交互。

-   **现有的 CSR 项目希望渐进式优化？** -> **CSR + RSC**
    -   *优势*：可以在不改变整体架构的情况下，将一些纯展示、数据密集的组件改造为 Server Components，逐步减轻客户端的负担。

-   **应用内部交互复杂，且对 SEO 和首屏无要求？** -> **CSR (客户端渲染)**
    -   *示例*：复杂的后台管理系统、在线编辑器、数据分析仪表盘。
    -   *优势*：开发模型简单直观，生态成熟。

### 组合使用

EdenX 的强大之处在于它允许你在同一个项目中**组合使用多种渲染模式**。

-   **SSG + SSR**：一个典型的组合。例如，一个电商网站的“关于我们”和“帮助中心”页面可以使用 SSG 生成，因为它们内容固定；而商品详情页则使用 SSR，因为价格和库存是动态变化的。
-   **Streaming SSR + RSC**：如前所述，这是性能优化的黄金组合。页面主体使用流式 SSR，而页面中非交互的、数据驱动的部分（如商品推荐列表、用户信息面板）可以作为 Server Components。

## 如何在 EdenX 中配置

在 EdenX 中切换和配置渲染模式非常简单，主要通过 `edenx.config.ts` 文件中的 `server` 和 `output` 配置项来控制。

### 启用 SSR

要启用 SSR，只需在配置中设置 `server.ssr = true`。

```ts
// edenx.config.ts
export default {
  server: {
    ssr: true, // 启用 SSR，默认为流式
  },
};
```

如果需要关闭流式渲染，回退到传统的字符串渲染模式（不推荐），可以这样配置：

```ts
// edenx.config.ts
export default {
  server: {
    ssr: {
      stream: false,
    },
  },
};
```

### 启用 SSG

启用 SSG 需要在 `output` 配置中设置 `ssg: true`。EdenX 会在执行 `pnpm run build` 命令时，为项目中的每个路由生成对应的 HTML 文件。

```ts
// edenx.config.ts
export default {
  output: {
    ssg: true,
  },
};
```

你还可以通过传递一个对象来自定义 SSG 的行为，例如只为特定的路由生成静态页面：

```ts
// edenx.config.ts
export default {
  output: {
    ssg: {
      '/about': {},
      '/help': {},
    },
  },
};
```

## 常见问题与最佳实践 (FAQ)

**Q1: 在 SSR/SSG 模式下，如何处理只在浏览器中存在的 API（如 `window`, `localStorage`）？**

**A1**: 服务端（Node.js 环境）没有 `window` 等浏览器特有的全局对象。在 SSR/SSG 组件中直接访问它们会导致构建或运行时错误。正确的处理方式是：
1.  **使用 `useEffect`**：将依赖浏览器 API 的代码包裹在 `useEffect` Hook 中，因为它只会在客户端执行。
    ```tsx
    useEffect(() => {
      // 这段代码只在客户端运行
      const theme = localStorage.getItem('theme');
      console.log(theme);
    }, []);
    ```
2.  **条件判断**：在代码执行前检查当前环境。
    ```tsx
    if (typeof window !== 'undefined') {
      // 客户端代码
    }
    ```
3.  **动态导入**：对于只在客户端使用的组件，可以使用动态 `import()` 结合 `React.lazy`，并设置 `ssr: false`。
    ```tsx
    import dynamic from 'next/dynamic';

    const ClientOnlyComponent = dynamic(
      () => import('../components/ClientOnlyComponent'),
      { ssr: false }
    );
    ```

**Q2: 为什么我的 SSR 页面在开发模式下正常，但生产环境构建失败/报错？**

**A2**: 这个问题通常是由于第三方库的兼容性问题引起的。一些老的库可能没有很好地支持 ESM（ECMAScript Modules）或在内部直接依赖了浏览器 API。排查步骤：
1.  **检查错误信息**：查看构建日志，定位是哪个库或文件导致了问题。
2.  **配置 `server.ssrExternals`**：如果某个库不应该被打包进服务端 bundle（因为它应该是 Node.js 环境的外部依赖），可以将其添加到 `ssrExternals` 列表中。
3.  **查找替代库**：寻找一个对 SSR 更友好的现代化替代品。
4.  **联系库作者**：如果可能，向库的维护者报告问题。

**Q3: SSG 和预渲染（Prerendering）有什么区别？**

**A3**: 在 EdenX 的语境下，SSG 是一个更广义的概念。你可以认为 SSG 就是在构建时对整个应用或部分路由进行“预渲染”。SSG 的目标是生成可以直接部署的静态 HTML 文件，而“预渲染”这个词有时也指在 CSR 应用中对首屏进行静态化处理以改善 SEO 的一种技术。在 EdenX 中，`output.ssg` 是实现这一目标的主要配置方式。

**Q4: 我应该在什么时候考虑引入 RSC？它会增加开发复杂性吗？**

**A4**: 当你发现应用的客户端 JavaScript 体积过大，导致加载和交互缓慢，并且应用中有大量纯展示、数据密集型的组件时，就是引入 RSC 的好时机。RSC 的引入确实会带来新的心智模型，你需要明确区分“服务端组件”（Server Components）和“客户端组件”（Client Components，即我们通常编写的带 `useState`, `useEffect` 的组件）。初期可能会增加一些学习成本，但它带来的性能收益对于追求极致体验的应用是巨大的。可以从一些独立的、简单的组件开始尝试，逐步感受其优势。
