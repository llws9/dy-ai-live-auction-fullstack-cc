# kb-docs-validator

> Note: All `python3 scripts/...` commands below must be executed with the current skill directory as the working directory (the directory containing this SKILL.md).

## What This Skill Does

- Scan one or more doc roots and **detect conflicts** among knowledge-doc files:
  - `CLAUDE.md` vs `docs/*` references
  - `docs/*` vs legacy root files (`interface.md/domain.md/workflow.md/rule.md`)
  - `AGENTS.md` link state (`AGENTS.md` should point to `CLAUDE.md`)
- Use the **codebase as the single source of truth** to validate key claims in docs (entry headers, key classes).
- Produce a **human-reviewable report** and, when safe, apply **minimal** doc fixes.
- When truth cannot be derived from code unambiguously, **ask for human confirmation/correction** before editing.
- Validate `docs/rule.md` against the shared rule contract: `references/kb-rule-quality-contract.md`.

## How to Use

### 1) Run the validator script

```bash
python3 scripts/validate_kb_docs.py <doc_root1> [doc_root2 ...]
```

### 2) Resolve conflicts (workflow)

1. **Classify conflicts** from the report:
  - Layout/duplication: legacy root docs vs `docs/` docs
  - Broken references: CLAUDE/README links point to non-existent targets
  - Content mismatch: docs claim entry points / key classes that don’t match code
2. **Verify against code first**:
  - Prefer public headers under `ModuleInterface/Interface/`.
  - Prefer actual service registration/entry points (`*ModuleService`, `LazyRegister`, boot tasks, etc.)
3. **Fix docs minimally**:
  - Update links and file paths.
  - Consolidate duplicates (usually pick `docs/` as canonical; keep root files as redirects if needed).
4. **Ask human when needed**:
  - If two doc variants both plausibly match code (e.g., differing domain narrative), ask which to keep.
  - If module ownership/feature scope is unclear, ask the owner.
5. **Run quality checks (principles-based)**:
  - Concise & maintainable:
    - Keep guidance concise; avoid “prompt bloat” and low-signal rules.
    - Keep content broadly applicable; move rare operations into module-level docs or a dedicated doc file.
    - Prefer iterative refinement; remove outdated guidance discovered during validation.
    - Avoid over-prescriptive process details and one-off steps.
  - Rule quality for `docs/rule.md`:
    - Use `references/kb-rule-quality-contract.md` as the canonical rule-quality policy.
    - The script enforces the executable subset: scenario blocks, `MUST` / `MUST NOT`, parent-rule duplication, scoped length budgets, misplaced inventories, generic phrases, table-based pseudo rules, low scenario density, and excessive code examples.
    - Scope classification is deterministic and explainable: child doc-root count first, aggregate path names second, then module layouts/code-file/subdirectory counts.
    - For existing low-quality `rule.md`, prefer minimal rewrite into the scenario format; ask before deleting large sections.
  - Checklist for `CLAUDE.md` changes:
    - Applicability: would ~90% of sessions benefit from each added/changed guideline?
    - Discoverability: does it include key paths and file/function names where appropriate?
    - Concision: remove outdated statements and avoid duplicating other docs under the same doc root.
6. **Ask human review when needed (quality gates)**:
  - If “Concise & maintainable” checks fail (too long, too specific, redundant), ask which parts to keep or remove.
  - If “Applicability/Discoverability” requires product/domain judgment, ask the doc owner to confirm intent.
  - If edits would remove/merge large sections or change doc structure, request human confirmation before applying.
