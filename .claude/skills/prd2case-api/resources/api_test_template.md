### 1.3 Global Context Variables

> 📝 **Instruction**: Declare **suite-level static variables only**. These variables may be copied into multiple per-API Go test files when `case.md` is converted to Go tests, so they must be safe for every consumer API.
> 
> ⚠️ **Global Variable Source Rules**: The source of a global variable **MUST** fall into one of the following three categories. They **CANNOT** be extracted from the response of a specific test case step. Local variables (used only in a single test case) should not be put here.
>
> ⚠️ **Cross-Test-Case Reference is FORBIDDEN**: A local variable extracted in one test case (e.g., TC-001) **CANNOT** be referenced in another test case (e.g., TC-002). If a runtime-extracted value is needed by another case, re-extract it in that case or declare an API/File Context Variable with a setup/helper strategy. Do **NOT** promote runtime-extracted values to this section.
>
> ⚠️ **Go Test Split Compatibility**: Later conversion may place the same API's cases in one Go file and different APIs in different files. Therefore, API-specific resource IDs, environment samples, setup results, and secrets must not be defined as suite globals. Put API-specific values in the API/File Context Variables section below.
>
> 1. **User Provided (Constant)**: Variables that MUST be manually provided by the user (e.g., specific test IDs, environment keys). You **MUST highlight** these values and sources using bold text and an explicit placeholder like **`[USER_INPUT_REQUIRED: <What to input>]`** to make them easily identifiable.
> 2. **Utility Function**: Dynamically generated values using built-in methods (e.g., `__getTimestamp()`).
> 3. **Model Generated (Constant)**: Fixed context values generated by the model during test design (e.g., a standard test tenant ID like `10086`).
>
> ✅ Suitable examples: tenant code shared by all APIs, non-secret enum constants, replay-safe timestamp/UUID helpers, non-existing sentinel IDs used only for negative tests.
> ❌ Unsuitable examples: tokens/cookies/auth headers, values extracted from another test case, existing resource IDs that should be found by List/Search setup, API-specific owner IDs, API-specific strategy/order/project IDs.

### \[TC-001] \[positive] \[Test Case Name] API Name / Core Scenario Name

**1. Test Case Information**

- **Test Type**: Single API / Scenario *(Choose one)*
- **Priority**: P0 / P1 / P2 *(P0 for core blockers, P2 for edge/exception cases)*
- **Data Dependencies & Local Variables**:
  > 📝 **Data Dependency Rules**: Provide the dependent data strictly following these priorities:
  > **a. Priority 1 (Knowledge Base)**: If the knowledge base provides the exact data or method, write it directly.
  > **b. Priority 2 (Auto Datagen)**: Write a requirement prompt tagged with `[tt-nova-datagen]` (e.g., `[tt-nova-datagen] Please generate an active SKU`). 
  > **c. Priority 3 (Manual Prompt)**: Tagged as `[Manual Prompt] <requirement>`.
  - *Data Dependency*: `[tt-nova-datagen] Please generate an active product SKU with stock > 0 for test_user_001.`
  - *Local Variables*: `{{LOCAL_SKU_ID}}` (Variables extracted from steps and scoped ONLY to this test case).

**2. Test Steps & Assertions**

**Step 1: \[Action Description] (HTTP Request Example)**

- **API Contract**: `[p.s.m]` -> `[Method] [Path]` [(View API)](#)
- **Protocol**: HTTP
- **Request Parameters**:
  - **Headers**:
    - `X-Device-Type`: `iOS`
  - **Path Parameters** *(Omit if none)*:
    - `item_id`: `"123456"`
  - **Query Parameters** *(Omit if none)*:
    - `keyword`: `"apple"`
    - `page_size`: `20`
  - **Body Parameters** *(Omit for GET requests)*:
    ```json
    {
      "category_id": 9876,
      "timestamp": "{{CURRENT_TIMESTAMP}}"
    }
    ```
- **Assertions**:
  > 📝 **Assertion Writing Rules**:
  > Every step **MUST** include `[Status]` and `[Business]` assertions as the baseline.
  > In addition, you must categorize your detailed field assertions into the following three types directly within the test step:
  > 
  > - **\[Scheme Change] (方案改动点字段)**: Fields that are newly added, renamed, type-changed, or structurally modified in the current requirement. These are the primary validation targets.
  > - **\[Business Key] (业务关键字段)**: Fields that are critical for business correctness (e.g., money, status, identity). If they are wrong, the core flow fails.
  > - **\[High-Volatility] (高波动字段)**: Optional and conditional fields. Assertions for these fields **MUST ONLY be executed if the field actually exists** in the response. Use conditional boundary, type, or state-machine assertions instead of exact-value matching.
  >
  > **Format**: `[Category] <field_path> <operator> <expected_value> // <reason>`
  - \[Status] `StatusCode == 200`
  - \[Business] `$.data.items.length > 0` *(Validation: Can correctly retrieve existing items)*
  - \[Scheme Change] `$.data.items[0].sku_id IS NOT NULL` *(NEW field — must exist)*
  - \[Scheme Change] `$.data.items[0].promotion_tag IN ["HOT","NEW","SALE"]` *(MODIFIED — enum constraint)*
  - \[Business Key] `$.data.items[0].price > 0` *(Pricing — prevents zero/negative charge)*
  - \[Business Key] `$.data.items[0].currency == "USD"` *(Pricing — locale currency match)*
  - \[High-Volatility] `IF $.data.items[0].stock EXISTS THEN $.data.items[0].stock >= 0` *(Boundary — optional/conditional field check)*
  - \[High-Volatility] `IF $.data.items[0].cover_url EXISTS THEN $.data.items[0].cover_url matches "^https?://"` *(Format — valid URL check if field exists)*
- **Variable Extraction**: *(Ignore if no extraction is needed)*
  - Extract `$.data.items[0].sku_id` and assign it to the local variable `{{LOCAL_SKU_ID}}`.

**Step 2: \[Action Description] (RPC Request Example)**
