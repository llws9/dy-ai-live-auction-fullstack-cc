## User Scenarios & Testing *(mandatory)*

### User Story 1 - 实时排名同步 (Priority: P1)

**角色**：用户（竞拍参与者）

**描述**：用户在竞拍过程中，当有人出价时，系统能够实时更新排名列表，所有参与者都能看到最新的排名变化，提升竞拍体验和参与感。

**为什么这个优先级**：实时排名同步是竞拍核心体验的关键部分，直接影响用户的参与感和竞争意识。没有实时排名，用户无法及时了解自己的竞争位置，降低了竞拍的紧张感和趣味性。

**技术实现**：

**涉及服务**：
- `auction-service` (`backend/auction`) - 竞拍服务

**后端变更**：
| 变更类型 | 文件/方法 | 描述 |
| --- | --- | --- |
| 修改 | `service/bid.go#PlaceBid` | 出价成功后调用排名广播 |
| 新增 | `service/bid.go#broadcastRanking` | 查询最新排名并广播到WebSocket |
| 修改 | `websocket/message.go` | 新增 `rank_update` 消息类型 |

**前端变更**：
| 变更类型 | 组件/页面 | 文件路径 | 描述 |
| --- | --- | --- | --- |
| 修改 | WebSocket服务 | `services/websocket.ts` | 处理 `rank_update` 消息 |
| 修改 | 竞拍页面 | `pages/Auction/index.tsx` | 接收并显示排名更新 |
| 修改 | 排名组件 | `pages/Auction/Ranking.tsx` | 实时排名展示 |

**调用链**：
```
用户出价 → BidService.PlaceBid → broadcastRanking → WebSocket Hub → 广播到所有客户端 → 前端更新排名
```

**独立测试**：可以通过模拟多个用户同时出价，验证排名是否实时更新到所有客户端来独立测试。

**验收场景**：
1. **Given** 用户A、B、C正在参与竞拍，**When** 用户A出价100元，**Then** 用户B、C在200ms内收到排名更新通知
2. **Given** 排名列表显示前10名，**When** 新用户出价并进入前10，**Then** 排名列表实时更新
3. **Given** 100人同时出价，**When** 并发出价请求，**Then** 排名同步正常，无数据错乱

### User Story 2 - 断线重连机制 (Priority: P1)

**角色**：用户（竞拍参与者）

**描述**：当用户在网络波动导致WebSocket连接断开时，系统能够自动检测并使用指数退避策略进行重连，重连成功后自动同步最新的竞拍状态，确保用户不会错过重要的竞拍信息。

**为什么这个优先级**：网络不稳定是移动端用户的常见问题，没有自动重连机制会导致用户错过出价机会，严重影响用户体验和竞拍公平性。

**技术实现**：

**涉及服务**：
- `auction-service` (`backend/auction`) - 竞拍服务

**后端变更**：
| 变更类型 | 文件/方法 | 描述 |
| --- | --- | --- |
| 修改 | `websocket/client.go#ReadPump` | 完善心跳检测和pong处理 |
| 新增 | `websocket/client.go#handleReconnect` | 处理客户端重连请求 |
| 修改 | `websocket/room.go` | 支持客户端重新加入房间 |
| 新增 | `websocket/state_sync.go` | 重连后状态同步逻辑 |

**前端变更**：
| 变更类型 | 组件/页面 | 文件路径 | 描述 |
| --- | --- | --- | --- |
| 修改 | WebSocket服务 | `services/websocket.ts` | 实现自动重连逻辑 |
| 新增 | 重连Hook | `hooks/useReconnect.ts` | 指数退避策略实现 |
| 修改 | 竞拍页面 | `pages/Auction/index.tsx` | 重连后同步最新状态 |

**指数退避策略**：
- 第1次重连：等待1秒
- 第2次重连：等待2秒
- 第3次重连：等待4秒
- 第4次重连：等待8秒
- 最大等待：30秒
- 最大重试次数：10次

**交互流程**：
```
断线检测 → 自动重连（指数退避：1s→2s→4s→8s→max 30s）→ 重连成功 → 同步最新状态
```

**独立测试**：可以通过手动关闭WebSocket连接，验证系统是否自动重连并恢复状态来独立测试。

**验收场景**：
1. **Given** 用户正在竞拍页面，**When** 网络断开10秒后恢复，**Then** 系统自动重连成功，状态同步准确
2. **Given** 用户已重连5次失败，**When** 第6次尝试，**Then** 继续使用指数退避策略重连
3. **Given** 用户断线期间有新的出价，**When** 重连成功，**Then** 用户能看到最新的价格和排名

**为什么这个优先级**：管理后台是主播/商家运营竞拍业务的核心工具，没有管理后台将无法有效地管理商品和竞拍活动，直接影响业务运营效率。

### Edge Cases

1. **WebSocket连接超限**：当同时在线用户超过系统限制时，如何处理新连接请求？
2. **重连失败超过最大次数**：当用户重连10次后仍失败，如何提示用户并提供备选方案？
3. **并发创建商品冲突**：多个主播同时创建同名商品时，如何处理唯一性约束？
4. **动画性能影响**：低端设备上动画可能导致卡顿，如何降级处理？
5. **时间同步失败**：服务端时间同步消息丢失时，如何保证倒计时准确性？
6. **历史记录查询性能**：用户参与大量竞拍后，历史记录查询可能变慢，如何优化？

## Requirements *(mandatory)*

### Functional Requirements

**实时排名同步**：
- **FR-001**: 系统 MUST 在出价成功后实时广播排名更新到所有参与用户
- **FR-002**: 系统 MUST 在200ms内完成排名广播（从出价成功到用户收到更新）
- **FR-003**: 系统 MUST 支持至少100个用户同时在线时排名同步正常

**断线重连机制**：
- **FR-004**: 系统 MUST 自动检测WebSocket连接断开
- **FR-005**: 系统 MUST 使用指数退避策略自动重连（最大重试10次）
- **FR-006**: 系统 MUST 在重连成功后自动同步最新竞拍状态
- **FR-007**: 系统 MUST 保证重连成功率 > 99%

**PC管理后台**：
- **FR-008**: 系统 MUST 提供商品管理功能（创建、编辑、删除、搜索）
- **FR-009**: 系统 MUST 提供竞拍管理功能（列表查看、状态管理、取消操作）
- **FR-010**: 系统 MUST 提供订单管理功能（列表查看、支付状态、发货管理）
- **FR-011**: 系统 MUST 支持配置竞拍规则（起拍价、加价幅度、封顶价、时长、延时机制）

**体验优化功能**：
- **FR-012**: 系统 MUST 提供关键场景动画反馈（出价成功、延时触发、竞拍结束）
- **FR-013**: 系统 MUST 提供毫秒级倒计时显示（误差 < 100ms）
- **FR-014**: 系统 MUST 提供用户历史记录查询功能
- **FR-015**: 系统 MUST 支持服务端时间同步机制

### Key Entities

- **竞拍排名（Ranking）**：表示竞拍中的排名信息，包含用户ID、出价金额、排名位置、出价时间
- **WebSocket连接状态（ConnectionState）**：表示WebSocket连接的状态，包含连接状态、重连次数、最后活动时间
- **商品（Product）**：表示竞拍商品，包含名称、描述、图片、状态
- **竞拍规则（AuctionRule）**：表示竞拍规则配置，包含起拍价、加价幅度、封顶价、时长、延时配置
- **订单（Order）**：表示竞拍成交后的订单，包含商品信息、中标者、成交价格、支付状态、发货状态
- **历史记录（History）**：表示用户的竞拍历史，包含参与的竞拍列表、中标记录、出价历史

## Risk Points

### 1. WebSocket并发性能

**风险描述**：排名广播时大量并发消息可能导致性能瓶颈

**影响范围**：auction-service WebSocket Hub

**缓解措施**：
- 实现消息节流机制（throttle.go）
- 批量广播优化（每200ms最多推送一次）
- 消息队列缓冲
- 压力测试验证性能

---

### 2. 断线重连状态同步

**风险描述**：重连后需要同步最新状态，数据一致性挑战

**影响范围**：前端状态管理、后端状态缓存

**缓解措施**：
- 服务端缓存最新竞拍状态
- 重连时主动推送最新状态
- 前端状态版本号校验
- 幂等性设计防止重复更新

---

### 3. 前后端接口一致性

**风险描述**：并行开发可能导致接口不匹配

**影响范围**：所有API接口

**缓解措施**：
- 先定义接口文档，前后端共同遵循
- 使用TypeScript类型定义，接口类型共享
- API Mock服务器提前联调
- 自动化接口测试

---

### 4. 管理后台权限控制

**风险描述**：缺少用户权限验证，可能导致安全风险

**影响范围**：admin前端、gateway权限中间件

**缓解措施**：
- 实现JWT认证中间件
- 角色权限控制（RBAC）
- 敏感操作审计日志
- API访问频率限制

---

### 5. 动画性能影响

**风险描述**：低端设备上动画可能导致性能问题

**影响范围**：前端用户体验

**缓解措施**：
- 动画性能监控
- 自动降级策略（检测FPS）
- CSS硬件加速
- 可选关闭动画设置

---
